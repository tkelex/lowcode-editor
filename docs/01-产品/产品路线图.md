# 产品路线图

## 目标

把当前低代码编辑器继续推进成可上线、可维护、可扩展的低代码平台雏形。后续规划不再按聊天过程拆散，而是按产品阶段维护：先保证上线和主链路稳定，再补协作、资产、数据源和更完整的平台治理。

## 当前已具备

- 注册、登录、项目管理、页面管理。
- 编辑器搭建、预览、保存、再次打开恢复。
- 页面版本历史、回滚、删除版本记录。
- 页面发布、公开访问、取消发布。
- 项目成员权限：owner、editor、viewer。
- 项目与页面操作审计。
- 平台管理员后台：用户、项目、发布页和全局审计治理。
- 事件动作系统、P3 常用物料、源码面板、响应式画布、大纲树和右键菜单。
- 共享 schema 包、旧 schema 迁移、保存/发布前校验。
- CI、E2E、API smoke、Docker/Nginx 模板、生产环境变量校验、备份恢复脚本和上线验收清单。

## 规划原则

- 每个功能都同时考虑编辑态、预览态、保存恢复、发布态和旧数据迁移。
- 涉及事件、schema、权限、发布页的改动必须补回归验证。
- 不为了“企业级”引入过重架构；优先保持单体全栈项目清晰可维护。
- 新增物料必须补齐 dev、prod、默认 props、setter、events、methods、schema registry 和验证路径。
- 每个阶段完成后同步文档中心、上下文索引和复盘记录。

## 阶段 A：上线演练与交付收口

目标：把“代码和文档层面可上线”推进到“真实环境演练通过”。

任务：

- 配置真实域名和 HTTPS。
- 使用生产环境变量启动 Docker Compose 栈。
- 执行数据库备份和恢复演练。
- 在部署环境运行健康检查和 API smoke。
- 按上线验收清单手测注册、登录、保存、发布、公开页、管理员治理。
- 记录上线版本、备份文件、验收结果和回滚预案。

验收：

- 部署环境 `GET /api/health` 正常。
- API smoke 通过。
- 上线验收清单全部通过。
- 至少有一个可用管理员账号。
- 出现异常时能根据 `requestId` 查日志并执行回滚。

## 阶段 B：协作与发布治理

目标：让多人协作和发布管理更接近真实团队使用。

任务：

- 完善成员管理前端 UI：成员列表、角色修改、移除成员、权限按钮禁用。
- 完善项目审计前端 UI：按动作、页面、时间筛选。
- 增加版本差异对比：先做 JSON diff，再逐步做组件树语义 diff。
- 增强发布管理：发布记录、发布版本回滚、发布路径管理。
- 预留自定义域名模型和审核流程。

验收：

- viewer 只能浏览，不能保存、发布或管理成员。
- editor 可编辑页面，但不能管理成员和项目设置。
- owner 能管理成员并查看审计。
- 发布前能看清目标版本和当前草稿的差异。

## 阶段 C：模板与素材资产

目标：让页面搭建不只依赖前端内置模板和手填图片 URL。

任务：

- 新增页面模板和区块模板后端存储。
- 支持从当前页面或选中组件保存为模板。
- 支持模板分类、标签、封面、创建人和可见性。
- 新增素材管理：图片上传、文件类型和大小限制、素材列表、删除和审计。
- 生产环境逐步把本地上传目录切换为对象存储。

验收：

- 用户可以保存常用表单、表格、详情区块为模板。
- 新建页面时可以从模板生成初始 schema。
- Image 等物料可以从素材库选择图片。
- 发布页能稳定访问素材。

## 阶段 D：数据源与动态页面

目标：让组件可以绑定接口、变量和表达式，形成更真实的业务页面。

任务：

- 完善页面级数据源管理：REST API、静态 JSON、后续数据库查询。
- 支持请求方法、URL、headers、参数和分页字段映射。
- Table、List、Select 等组件支持绑定数据源。
- 完善页面变量和表达式绑定。
- HTTP 请求动作、表单提交结果和数据源响应可写入页面变量。

验收：

- 表格可配置接口并渲染远程数据。
- 一个输入框变化可以驱动文本、表格查询参数或按钮状态。
- 请求失败有明确错误提示，不导致公开页白屏。

## 阶段 E：工程质量与性能

目标：降低长期维护成本，提升生产体验。

任务：

- 拆分生产大 chunk：Monaco、编辑器设置面板、图表、Ant Design 依赖按需加载。
- 发布页运行态避免加载编辑器专用模块。
- CI 增加更完整的数据库迁移、API smoke 和 E2E 报告。
- 接入日志采集、错误上报和基础告警。
- 增加文档链接检查和脚本化验收。

验收：

- 发布页首屏不加载 Monaco 和编辑态设置面板。
- CI 能拦截构建、迁移、权限和发布主链路回归。
- 线上错误能被定位到请求、页面或事件动作。

## 近期推荐顺序

1. 先完成真实部署演练，并把结果补到复盘文档。
2. 再做成员管理 UI、审计 UI 和版本差异对比。
3. 继续推进模板后端存储和素材管理。
4. 最后进入数据源、变量和表达式的深水区。

## 不建议现在做

- 不建议马上改成复杂微服务或多仓架构。
- 不建议一次性迁移到 `apps/web`、`apps/server`，当前脚本和部署已经稳定。
- 不建议在没有测试保护时重写编辑器渲染核心。
- 不建议开放公网注册给不可信用户，除非已经完成备份、监控、安全和管理员值守。
