# 企业级低代码平台建设规划

本文用于把当前低代码页面编辑器逐步升级为类似 JeecgBoot 的企业级低代码应用开发平台。它是后续多 AI 对话框并行开发的产品、技术和验收基准。

## 1. 目标定位

当前项目已经具备页面搭建、schema 保存、预览、发布、版本、模板、项目成员和管理员后台能力。下一阶段目标不是继续单纯堆 UI 组件，而是补齐企业应用平台的核心链路：

```text
业务模型设计 -> 自动生成 CRUD 能力 -> 页面/菜单/权限挂载 -> 数据驱动组件 -> 发布和运维治理
```

最终平台应支持不会写代码的用户通过配置完成常见后台系统，例如客户管理、订单管理、商品管理、审批表单、报表看板等。

## 2. 技术栈基线

继续沿用当前技术栈，避免为了追赶大型平台而重写。

| 层级 | 当前技术 | 规划原则 |
| --- | --- | --- |
| 前端框架 | Vite + React + TypeScript | 保持单页应用，新增业务模块放在 `src/features/*` |
| UI | Ant Design + Tailwind CSS | 后台管理、表格、表单优先用 Ant Design |
| 编辑器状态 | Zustand persist | 组件树继续由 `useComponetsStore` 管理 |
| 拖拽 | React DnD | 编辑器拖拽不重写，按物料能力扩展 |
| 代码编辑 | Monaco Editor | schema、表达式、SQL/JSON 配置复用 |
| 后端 | NestJS + TypeScript | 新能力按 Nest module 拆分 |
| ORM/数据库 | Prisma + PostgreSQL | 优先用 Prisma 管理平台元数据，动态业务数据谨慎设计 |
| 共享协议 | `packages/lowcode-schema` | schema、动作、模型协议放共享包 |
| 鉴权 | JWT + Passport | 扩展角色、菜单、按钮、数据权限 |
| 部署 | Docker Compose + Nginx | 继续维护前端、后端、PostgreSQL 三件套 |

## 3. 总体架构演进

平台需要从“页面 schema 驱动”升级到“应用元数据驱动”。

```text
User / Role / Org / Permission
  -> App / Project
  -> Menu
  -> Page Schema
  -> Data Model
  -> Dynamic CRUD Runtime
  -> Workflow / Report / AI Generator
```

新增能力必须遵守两个边界：

- 平台元数据：用户、角色、菜单、模型、字段、页面、模板、发布记录等，存储在 Prisma 管理的 PostgreSQL 表中。
- 用户业务数据：由动态模型产生的数据，第一阶段可存 JSONB 通用表，后续再评估是否生成真实业务表。

## 4. 阶段路线图

### P0：稳定当前编辑器

目标：把现有 MVP 修到可信，避免在 bug 地基上继续加复杂能力。

范围：

- 修复编辑器拖拽、选择、属性、样式、事件、预览、发布的已知问题。
- 补齐核心回归测试。
- 确认公开发布页不能执行 customJS。
- 保持 schema 迁移和旧页面兼容。

验收：

- `npm run check` 通过。
- `npm run test:e2e:editor` 通过。
- 手动完成“创建页面 -> 拖组件 -> 配事件 -> 保存 -> 刷新恢复 -> 发布访问”。

### P1：菜单与应用壳

目标：发布内容不只是 `/publish/:publicId`，还可以挂到系统菜单中，形成内部业务应用。

范围：

- 菜单管理：树结构、排序、图标、路径、状态、关联页面。
- 角色菜单权限：角色可见哪些菜单。
- 页面挂载菜单：Page 可以作为内部菜单页面打开。
- 应用布局：左侧菜单 + 顶部用户信息 + 内容区。

验收：

- 管理员可配置菜单。
- 普通用户只能看到授权菜单。
- 点击菜单能打开对应低代码页面。

### P2：数据模型设计器

目标：让用户能定义业务对象，例如客户、订单、商品。

范围：

- 模型列表：名称、编码、描述、状态。
- 字段设计器：字段名、字段编码、类型、必填、唯一、默认值、校验、控件类型。
- 字典管理：枚举值、颜色、排序、启用状态。
- 模型版本：字段变更需要记录版本和变更来源。
- 模型发布：草稿模型发布后才进入 CRUD 运行态。

字段类型首批支持：

- string、text、number、boolean、date、datetime、enum、json、image、file、user、org。

验收：

- 创建模型和字段后可以保存、编辑、发布。
- 字段编码合法性、重复字段、保留字有校验。
- 模型元数据有后端权限校验和审计日志。

### P3：动态 CRUD 运行时

目标：模型发布后，平台自动拥有增删改查能力。

首选方案：

- 平台元数据仍由 Prisma schema 管理。
- 动态业务数据先用通用表存储：

```text
DynamicRecord {
  id
  projectId
  modelId
  data Json
  createdById
  updatedById
  createdAt
  updatedAt
  deletedAt
}
```

原因：

- 不频繁改数据库结构，适合早期快速迭代。
- 避免多个 AI 同时改 `server/prisma/schema.prisma` 造成冲突。
- 后续稳定后再增加“生成真实表”的能力。

接口范围：

```text
GET    /api/models/:modelId/records
POST   /api/models/:modelId/records
GET    /api/models/:modelId/records/:recordId
PATCH  /api/models/:modelId/records/:recordId
DELETE /api/models/:modelId/records/:recordId
```

能力要求：

- 分页、排序、关键词搜索。
- 按字段查询。
- 新增、编辑、删除、批量删除。
- 字段校验。
- 创建人、更新时间等系统字段。
- 审计日志。

验收：

- 任意发布模型都能通过统一接口完成 CRUD。
- 非授权用户不能读写模型数据。
- 字段校验失败返回统一错误码。

### P4：模型驱动组件

目标：拖拽出来的页面能直接操作业务数据。

新增/升级物料：

- ModelTable：绑定模型，展示记录，支持分页、查询、排序、操作列。
- ModelForm：绑定模型，支持新增/编辑记录。
- SearchForm：根据模型字段生成查询条件。
- DetailView：展示单条记录详情。
- ImportExportButton：导入导出模型数据。

事件动作扩展：

- createRecord
- updateRecord
- deleteRecord
- queryRecords
- openRecordModal
- refreshModelTable

验收：

- 不写代码搭建一个“客户管理”页面。
- 可以新增客户、编辑客户、删除客户、查询客户。
- 页面刷新后数据仍存在。

### P5：CRUD 页面生成器

目标：用户选择一个模型后，一键生成标准后台页面。

生成内容：

- 页面 schema。
- 表格组件。
- 查询区。
- 新增/编辑弹窗。
- 删除确认。
- 菜单项。
- 默认按钮权限点。

运行模式：

- 零代码模式：生成 schema 并由运行时渲染。
- 低代码模式：后续可导出代码，给开发者二次开发。

验收：

- 选择模型后 30 秒内生成可用 CRUD 页面。
- 生成页面可继续在编辑器中调整。

### P6：权限增强

目标：从项目角色升级为企业后台常见权限体系。

范围：

- 用户管理。
- 角色管理。
- 组织/部门管理。
- 菜单权限。
- 按钮权限。
- 字段权限。
- 数据权限。

建议顺序：

1. 菜单权限。
2. 按钮权限。
3. 字段权限。
4. 数据权限。

验收：

- 同一页面中，不同角色看到不同按钮。
- 同一模型中，不同角色可读/可写字段不同。
- 数据权限至少支持“本人、部门、全部”。

### P7：代码生成器

目标：复杂业务可以从配置转成可维护源码。

生成目标：

- NestJS controller/service/dto。
- Prisma schema 片段或 SQL migration 草稿。
- 前端 API client。
- React 列表页、表单页、详情页。
- 菜单和权限 seed。

原则：

- 生成代码默认进入独立目录，不覆盖手写代码。
- 首期只生成草稿，不自动写入核心运行路径。
- 代码生成必须可重复、可 diff、可删除。

验收：

- 对一个模型生成完整 CRUD 草稿。
- 生成结果能通过 lint/build。

### P8：报表与大屏

目标：让业务数据可以被分析展示。

范围：

- 数据集配置。
- 表格报表。
- 图表组件。
- 指标卡片。
- 大屏画布。
- 导出 Excel/PDF。

首期只支持模型数据集，不开放任意 SQL。

### P9：工作流

目标：支持审批类业务。

范围：

- 流程模型。
- 节点设计。
- 审批人规则。
- 表单绑定流程。
- 待办、已办、审批记录。

首期只支持简单串行审批。

### P10：AI 生成

目标：用自然语言生成模型、页面、菜单和初始权限。

依赖：

- 模型设计器稳定。
- CRUD 运行时稳定。
- 页面生成器稳定。

首期能力：

- 用户输入业务描述。
- AI 生成模型草稿。
- AI 生成 CRUD 页面草稿。
- 用户确认后入库。

## 5. 模块拆分与所有权

多 AI 并行时按模块认领，尽量不要跨模块修改。

| 模块 | 作用 | 主要目录 | 依赖 | 常规测试 |
| --- | --- | --- | --- | --- |
| 权限与菜单 | 内部应用壳、菜单、角色授权 | `src/features/access`、`server/src/modules/access` | auth、projects | 后端单测、API smoke、菜单 e2e |
| 数据模型 | 模型/字段/字典元数据 | `src/features/models`、`server/src/modules/models`、`packages/lowcode-schema/src/model-*` | projects、audit | schema 单测、模型 API smoke |
| 动态记录 | 通用 CRUD 数据运行时 | `server/src/modules/dynamic-records`、`src/shared/api/dynamic-records.ts` | models、access | CRUD API smoke、权限测试 |
| 模型物料 | ModelTable、ModelForm 等 | `src/editor/materials/Model*`、`src/editor/registry/configs/data.tsx` | dynamic-records、editor runtime | runtime 测试、editor e2e |
| 页面生成器 | 模型生成页面 schema | `src/features/generator`、`server/src/modules/generator` | models、pages、menus | 生成器单测、端到端创建页面 |
| 代码生成 | 输出源码草稿 | `server/src/modules/codegen`、`scripts/codegen` | models、templates | 快照测试、lint/build |
| 报表大屏 | 数据集、图表、看板 | `src/features/reports`、`server/src/modules/reports` | models、dynamic-records | 图表渲染 e2e、数据集测试 |
| 工作流 | 审批流和任务 | `src/features/workflow`、`server/src/modules/workflow` | models、users、access | 流程状态机单测 |
| AI 生成 | 自然语言生成配置 | `src/features/ai`、`server/src/modules/ai` | models、generator | mock LLM 测试、人工验收 |

## 6. 数据库演进原则

- 每个 AI 对话框一次只负责一个 Prisma migration。
- 迁移文件名必须描述业务，例如 `add_lowcode_models`。
- 改 `server/prisma/schema.prisma` 时必须同步 DTO、API 文档和 smoke 测试。
- 动态业务数据早期优先 JSONB 通用表，减少频繁 DDL。
- 不允许直接删除已有字段；需要先弃用，再迁移，再清理。
- 所有后端写操作必须记录审计日志，除非文档说明豁免。

## 7. API 设计原则

- 前端 API 封装放 `src/shared/api/*`。
- 后端模块按 `controller/service/dto` 拆分。
- 统一使用 JWT 鉴权。
- 项目内资源必须校验项目成员身份。
- 管理员接口必须放独立 admin 或明确 guard。
- 错误响应走 `BusinessException` 和统一错误码。
- 新增 API 必须同步 `docs/03-接口/接口说明.md`。

## 8. 测试策略

按风险选择测试，不允许只靠手动点。

| 改动类型 | 最低验证 |
| --- | --- |
| 文档 | 不需要跑全量测试，检查链接和路径 |
| 纯前端展示 | `npm run lint`、必要时 `npm run build` |
| 编辑器行为 | `npm run test:e2e:editor` |
| schema/事件/运行时 | `npm run test` |
| 后端 API | `npm run build --prefix server`、新增 smoke 或 node test |
| Prisma migration | `npm run prisma:generate --prefix server`、本地迁移验证 |
| 发布链路 | `npm run smoke:api`、公开页手动验收 |
| 跨模块大改 | `npm run check` + 相关 e2e/smoke |

每个模块交付必须包含：

- 正向用例。
- 权限拒绝用例。
- 非法输入用例。
- 旧数据兼容说明。

## 9. 质量门禁

合并前必须回答：

- 是否改了 schema、API、Prisma、权限或发布链路？
- 是否同步相关文档？
- 是否有测试覆盖核心路径？
- 是否会影响公开发布页安全？
- 是否会影响已有页面 schema 迁移？
- 是否能回滚？

禁止：

- 绕过后端权限，只靠前端隐藏按钮。
- 在公开页执行 customJS。
- 直接让前端调用后端内部实现文件。
- 把新 API 写到旧 `src/api/*` 主逻辑中。
- 无测试地修改 `components.tsx`、`Preview`、schema migration、权限服务。

## 10. 推荐近期任务包

按下面顺序开多个 AI 对话框，每个对话框只拿一个任务包。

### Task A：当前编辑器稳定性审计

范围：

- 只读审计 `src/editor/*`。
- 列出已知 bug、复现步骤、风险等级、建议测试。
- 不直接改代码。

### Task B：菜单模型设计

范围：

- 设计 `Menu`、`RoleMenuPermission` 数据结构。
- 产出 Prisma migration 和后端 API 草案。
- 不碰编辑器。

### Task C：数据模型协议设计

范围：

- 设计 model/field/dictionary 共享类型。
- 新增 `packages/lowcode-schema/src/model-*`。
- 写纯函数校验测试。

### Task D：动态记录 CRUD 后端

范围：

- 依赖 Task C。
- 实现动态记录接口。
- 写 API smoke。

### Task E：模型管理前端

范围：

- 依赖 Task C 的 API 类型。
- 实现模型列表和字段设计器。
- 不碰编辑器 runtime。

### Task F：模型表格物料

范围：

- 依赖 Task D。
- 新增 ModelTable 物料。
- 只改 data 类 registry 和对应 material。

### Task G：CRUD 页面生成器

范围：

- 依赖 Task E/F。
- 根据模型生成页面 schema。
- 不改动态 CRUD 后端。

## 11. 里程碑定义

### M1：内部应用壳

- 菜单管理完成。
- 页面可挂菜单。
- 角色控制菜单可见性。

### M2：模型可建

- 模型和字段可配置。
- 字典可配置。
- 模型可发布。

### M3：数据可用

- 发布模型可 CRUD。
- 有权限校验和审计。
- 前端有模型数据管理页。

### M4：页面可生成

- 一键生成 CRUD 页面。
- 页面可继续拖拽编辑。
- 页面可挂菜单并给角色使用。

### M5：平台可交付

- 可以交付一个客户管理/订单管理演示应用。
- 有部署文档、验收清单、回滚策略。
- 核心链路有自动化测试。
