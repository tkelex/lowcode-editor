# 事件绑定系统设计

## 目标

统一所有物料组件的事件绑定方式，避免不同组件把事件配置散落在 `props.onClick`、`props.onChange`、`props.onFinish` 等字段里。

新方案参考 amis 的事件动作思路：组件通过统一的 `onEvent` 声明事件，事件下按顺序配置 `actions`，每个动作通过 `actionType` 和 `args` 表达。

参考资料：

- amis 事件动作文档：https://develop365.gitlab.io/amis/zh-CN/docs/concepts/event-action/
- amis 行为文档：https://develop365.gitlab.io/amis/zh-CN/docs/concepts/action/index.html

## 设计原则

- 事件配置统一存储在组件 `props.onEvent` 下。
- 物料注册表只声明“组件支持哪些事件”，不负责保存动作内容。
- 运行态由统一事件执行器负责注入事件 props 和执行动作。
- 新 schema 兼容旧 schema，旧页面中 `props.onClick.actions` 这类配置仍可运行。
- 公开发布页继续禁用 `custom` 动作，避免访客页面执行用户自定义 JS。

## 组件事件 Schema

组件节点中的事件配置统一放在：

```ts
component.props.onEvent
```

示例：

```json
{
  "id": 2,
  "name": "Button",
  "props": {
    "text": "提交",
    "onEvent": {
      "click": {
        "actions": [
          {
            "actionType": "toast",
            "args": {
              "msgType": "success",
              "msg": "提交成功"
            }
          },
          {
            "actionType": "url",
            "args": {
              "url": "/success"
            }
          }
        ]
      }
    }
  },
  "desc": "按钮"
}
```

## 物料事件声明

物料注册表 `src/editor/stores/component-config.tsx` 中的 `events` 只描述当前组件支持的事件：

```ts
events: [
  {
    name: 'click',
    label: '点击事件',
  },
]
```

`name` 是低代码 schema 中的事件名。运行态会自动转换成 React props：

```text
click       -> onClick
doubleClick -> onDoubleClick
change      -> onChange
pressEnter  -> onPressEnter
finish      -> onFinish
ok          -> onOk
cancel      -> onCancel
```

如果未来有无法自动推导的事件，可以在事件声明中增加 `propName`：

```ts
{
  name: 'submit',
  label: '提交事件',
  propName: 'onFinish',
}
```

## 动作类型

当前实现多类动作，覆盖常见事件编排能力。

### toast

展示消息提示。

```json
{
  "actionType": "toast",
  "args": {
    "msgType": "success",
    "msg": "操作成功"
  }
}
```

### url

跳转链接。

```json
{
  "actionType": "url",
  "args": {
    "url": "https://example.com",
    "blank": true
  }
}
```

运行时会在执行前统一处理链接：

- `https://example.com`、`mailto:test@example.com` 这类已经包含协议的地址会原样跳转。
- `/publish/demo` 这类以 `/` 开头的地址按站内路径跳转。
- `baidu.com`、`www.baidu.com/a` 这类未填写协议的外部地址会自动补全为 `https://baidu.com`、`https://www.baidu.com/a`，避免浏览器把它们当成 `localhost` 下的相对路径。

### componentAction

调用另一个组件暴露的方法，例如打开弹窗或提交表单。

```json
{
  "actionType": "componentAction",
  "componentId": 5,
  "args": {
    "method": "open"
  }
}
```

### componentControl

组件联动动作，提供常用联动的可视化配置，不需要用户手写 JSON：

- 显示 / 隐藏组件。
- 启用 / 禁用组件。
- 设置值 / 清空值。
- 打开 / 关闭弹窗。
- 提交 / 重置表单。

### confirm

显示确认弹窗，确认后继续执行内嵌动作，取消后默认中断后续动作。

确认动作中的“确认后执行”和“取消后执行”都支持完整动作列表，可排序、复制、启用/禁用。

### condition

按表达式分支执行动作。表达式可使用 `event`、`context`、`args`，例如：

```js
event.value === 'admin'
```

条件动作提供常用条件模板，例如 `event.value` 有值、`event.value` 等于某个值、`event.checked` 为 true、`event.values` 指定字段有值。真假分支都支持完整动作列表。

### http

发送 HTTP 请求。请求结果会写入 `event.httpResponse`，请求错误会写入 `event.httpError`，后续自定义 JS 或条件动作可以继续使用这些数据。

HTTP 动作支持：

- 鉴权方式：不携带鉴权、当前登录用户 Token、手动 Bearer Token。
- Headers / Body 模板：字符串字段可使用 `{{event.value}}`、`{{context.component.id}}` 这类模板。
- 响应字段映射：可把响应写入 `event.api.xxx` 这类自定义路径。
- 错误字段映射：可把错误写入 `event.api.xxxError` 这类自定义路径。

### setComponentProps / setComponentStyles

更新目标组件的属性或样式，适合做点击按钮后修改文本、隐藏/显示组件、切换样式等交互。

### custom

执行自定义 JS。仅编辑器预览态允许执行；公开发布页禁用。

```json
{
  "actionType": "custom",
  "args": {
    "script": "context.showMessage('hello')"
  }
}
```

自定义 JS 执行时可使用：

```ts
context.component
context.eventName
context.eventData
context.args
context.showMessage(text, type?)
context.doAction(action)
```

## 运行态上下文

事件触发时，运行态会为每个动作提供统一上下文：

```ts
{
  component,
  eventName,
  eventData,
  args,
  components,
  componentRefs,
  allowCustomJS
}
```

`eventData` 会尽量从原始事件参数中提取常用字段：

- DOM/Input 事件：`value`、`checked`
- Select/Switch/Form 等组件：`value`
- 原始参数：`args`

## 旧 Schema 兼容

旧事件配置类似：

```json
{
  "props": {
    "onClick": {
      "actions": [
        {
          "type": "showMessage",
          "config": {
            "type": "success",
            "text": "成功"
          }
        }
      ]
    }
  }
}
```

新运行态会读取并转换为：

```json
{
  "props": {
    "onEvent": {
      "click": {
        "actions": [
          {
            "actionType": "toast",
            "args": {
              "msgType": "success",
              "msg": "成功"
            }
          }
        ]
      }
    }
  }
}
```

右侧事件面板编辑旧事件时，会写入新的 `onEvent` 结构。旧字段可以留在 schema 中，运行态优先读取 `onEvent`。

## 后续扩展方向

- 增加条件执行，例如 `expression`。
- 增加确认弹窗动作。
- 增加请求接口动作。
- 增加设置组件状态或更新数据源动作。
- 增加动作编排能力，例如失败分支、异步等待、动作日志。
