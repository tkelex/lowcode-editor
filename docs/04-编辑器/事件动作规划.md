# 物料事件与动作规划

## 为什么需要重新规划

当前事件面板的问题不是“没有统一 schema”，而是缺少产品层规则：

- 不同组件应该暴露不同事件。
- 同一个组件的不同事件，能执行的动作也应该不同。
- 事件动作弹窗应该根据当前事件过滤动作，而不是所有事件都能添加所有动作。
- 事件产生的数据不同，例如 `click` 没有输入值，`change` 有 `value`，`finish` 有表单提交值。

因此事件系统要拆成两层：

```text
组件事件能力：这个组件能触发什么事件
事件动作能力：这个事件允许配置什么动作
```

## 网络参考结论

本规划主要参考两类资料：

- amis 的事件动作模型：通过 `onEvent` 绑定事件，通过 `actions` 顺序执行动作，动作使用 `actionType` 和 `args` 描述。amis 还把动作分为通用动作、组件动作、广播动作、自定义动作。
- Ant Design 官方组件 API：每个物料底层大多来自 AntD，事件能力应尽量贴近对应 AntD 组件，例如 Button 的 `onClick`、Input 的 `onChange/onPressEnter/onClear`、Form 的 `onFinish/onFinishFailed/onValuesChange`、Modal 的 `onOk/onCancel/afterOpenChange`、Table 的 `onRow/onChange` 等。

## 当前项目物料清单

当前物料区或注册表中已有这些组件：

```text
Page
Container / Card
Button / Text / Image / Divider
Alert / Modal
Input / Select / Switch / Form / FormItem
Table / TableColumn
```

规划时按三类组件处理：

| 类型 | 组件 | 事件策略 |
| --- | --- | --- |
| 可交互组件 | Button、Input、Select、Switch、Form、Modal、Alert | 优先跟随 AntD 官方事件 |
| 展示组件 | Text、Image、Divider、Card、Table | 保留业务上有意义的事件，不盲目暴露所有 DOM 事件 |
| 结构配置节点 | FormItem、TableColumn | 默认不单独绑定事件，由父组件 Form/Table 统一处理 |

## 事件配置目标

统一存储仍使用：

```ts
component.props.onEvent[eventName].actions
```

但物料注册表中的事件声明需要更完整：

```ts
events: [
  {
    name: 'click',
    label: '点击事件',
    propName: 'onClick',
    description: '用户点击组件时触发',
    eventDataSchema: ['args'],
    allowedActions: ['toast', 'url', 'componentAction', 'custom']
  }
]
```

其中：

- `name`：低代码事件名，保存到 schema。
- `label`：右侧面板展示文案。
- `propName`：运行态注入给 React/AntD 的事件 prop，可省略并自动推导。
- `eventDataSchema`：这个事件会产生哪些数据。
- `allowedActions`：这个事件下允许添加哪些动作。

## 页面变量与数据源

当前事件系统已经支持 `setVariable` 动作，用来把事件数据、固定 JSON 值或安全表达式结果写入页面运行态变量：

```json
{
  "actionType": "setVariable",
  "args": {
    "path": "form.keyword",
    "expression": "event.value"
  }
}
```

可用表达式上下文：

```text
event       当前事件数据，例如 event.value、event.values
context     当前组件和运行态信息
variables   页面变量
args        原始事件参数
```

组件属性可通过 `{{variables.form.keyword}}` 绑定变量。Page 组件负责配置初始 `variables` 和 `dataSources`，这些内容会随页面 schema 保存、版本化和发布。

## 动作分类

动作池参考 amis 的动作编排方式，但结合当前项目先做轻量版本。动作分为四层：

```text
P0 当前可落地：复用已有能力，先把事件面板做顺
P1 产品增强：让表单、表格、组件状态真正联动
P2 高级编排：条件、异步、分支、全局广播
P3 安全增强：沙箱、权限、动作审计
```

### P0 先实现动作

这些动作能覆盖当前项目已有能力，也是第一阶段最适合落地的范围。

| actionType | 名称 | 说明 | 适用场景 |
| --- | --- | --- | --- |
| `toast` | 消息提示 | 展示成功、失败、警告、普通消息 | 所有交互事件 |
| `url` | 页面跳转 | 当前窗口或新窗口打开链接 | 点击类、提交成功类事件 |
| `componentAction` | 调用组件方法 | 调用弹窗打开/关闭、表单提交等方法 | 点击按钮、表单提交、弹窗事件 |
| `custom` | 自定义 JS | 执行用户脚本，编辑器预览可用，公开页禁用 | 开发者调试和高级扩展 |

### P1 后续动作

这些动作等 P0 稳定后再加。

| actionType | 名称 | 说明 |
| --- | --- | --- |
| `setVisible` | 显示/隐藏组件 | 控制目标组件显示状态 |
| `setDisabled` | 禁用/启用组件 | 控制表单组件或按钮禁用状态 |
| `setProps` | 修改组件属性 | 修改目标组件 props，例如设置文本、禁用按钮 |
| `setValue` | 设置表单值 | 设置 Input/Select/Switch/Form 的值 |
| `request` | 发送请求 | 调后端接口，把返回值放入上下文 |
| `confirm` | 二次确认 | 先弹确认框，再执行后续动作 |
| `copy` | 复制文本 | 复制固定文本或事件数据 |
| `reload` | 刷新数据 | 刷新 Table 数据源或未来数据源组件 |

### P2 高级动作

这些动作更接近完整低代码平台能力，可以等数据源、变量和状态模型稳定后再做。

| actionType | 名称 | 说明 |
| --- | --- | --- |
| `setState` | 设置页面变量 | 写入页面级变量，例如搜索条件、临时状态 |
| `broadcast` | 广播事件 | 通知其它组件响应某个自定义事件 |
| `wait` | 等待 | 延迟执行后续动作 |
| `condition` | 条件分支 | 根据表达式选择执行分支 |
| `loop` | 循环动作 | 对数组数据重复执行动作 |
| `download` | 下载文件 | 触发文件下载 |
| `print` | 打印页面 | 调用浏览器打印或局部打印 |
| `historyBack` | 返回上一页 | 调用浏览器历史返回 |
| `openDrawer` | 打开抽屉 | 如果后续新增 Drawer 组件，可作为组件动作 |

## 动作与事件数据的关系

事件动作不是只靠用户手写固定值，后续应该能引用事件数据。

示例：

```text
Input.change 产生 event.value
Select.change 产生 event.value / event.option
Switch.change 产生 event.checked
Form.finish 产生 event.values
Table.rowClick 产生 event.record / event.index
```

动作配置中可以预留表达式能力：

```json
{
  "actionType": "toast",
  "args": {
    "msg": "当前值是 ${event.value}"
  }
}
```

P0 可以先不实现表达式解析，但 schema 设计要留空间。

## 事件类型与动作规则

为了避免每个事件都能乱配动作，先定义事件类别。

| 事件类别 | 常见事件 | 事件数据 | 默认允许动作 |
| --- | --- | --- | --- |
| `ui` | `click`、`doubleClick` | 原始事件参数 | `toast`、`url`、`componentAction`、`custom` |
| `value` | `change`、`blur`、`pressEnter` | `value`、`checked`、原始参数 | `toast`、`componentAction`、`custom` |
| `submit` | `finish` | `values`、原始参数 | `toast`、`url`、`componentAction`、`custom` |
| `overlay` | `ok`、`cancel`、`close` | 原始事件参数 | `toast`、`componentAction`、`custom` |
| `lifecycle` | `mount`、`load`、`error` | 组件状态或资源状态 | `toast`、`componentAction`、`custom` |

规则：

- 点击类事件可以跳转链接。
- 输入值变化类事件默认不建议直接跳转，避免用户每输入一个字就跳走。
- 表单提交成功可以跳转，也可以弹消息、打开弹窗、调用组件方法。
- 弹窗确认/取消/关闭不建议跳转作为默认动作，但可以通过自定义 JS 扩展。
- 公开发布页不执行 `custom`。

## 物料事件规划

### Page

| 事件名 | 文案 | 事件类别 | 事件数据 | 允许动作 |
| --- | --- | --- | --- | --- |
| `mount` | 页面加载 | `lifecycle` | 页面组件、当前 schema | `toast`、`componentAction`、`custom` |
| `unmount` | 页面卸载 | `lifecycle` | 页面组件、当前 schema | `custom` |

说明：

- `Page` 不建议暴露点击事件。页面根节点太大，点击冒泡容易误触发。
- `mount` 需要运行态补 `useEffect` 或统一生命周期触发器，属于 P1。
- `unmount` 主要用于清理副作用，当前项目暂不急。

### Container

| 事件名 | 文案 | 事件类别 | 事件数据 | 允许动作 |
| --- | --- | --- | --- | --- |
| `click` | 点击事件 | `ui` | 原始事件参数 | `toast`、`url`、`componentAction`、`custom` |
| `doubleClick` | 双击事件 | `ui` | 原始事件参数 | `toast`、`componentAction`、`custom` |
| `mouseEnter` | 鼠标移入 | `ui` | 原始事件参数 | `toast`、`componentAction`、`custom` |
| `mouseLeave` | 鼠标移出 | `ui` | 原始事件参数 | `toast`、`componentAction`、`custom` |

说明：

- Container 是布局容器，点击事件可保留。
- 后续如果点击误触发明显，可以只让用户显式开启事件冒泡或阻止冒泡。
- mouseEnter/mouseLeave 放 P1，适合做悬浮提示或交互面板。

### Card

| 事件名 | 文案 | 事件类别 | 事件数据 | 允许动作 |
| --- | --- | --- | --- | --- |
| `click` | 点击事件 | `ui` | 原始事件参数 | `toast`、`url`、`componentAction`、`custom` |
| `doubleClick` | 双击事件 | `ui` | 原始事件参数 | `toast`、`componentAction`、`custom` |
| `tabChange` | 页签切换 | `value` | `key` | `toast`、`componentAction`、`custom` |

说明：

- Card 和 Container 类似，但更偏业务区块。
- 如果 Card 内部有 Button，运行态要注意事件冒泡导致 Card click 一起触发。
- 当前 Card 没有开启 tabs，tabChange 是 P1/P2 候选。

| `finishFailed` | 提交失败 | `submit` | `errorFields`、原始事件参数 | `toast`、`componentAction`、`custom` |
| `valuesChange` | 表单值变化 | `value` | `changedValues`、`allValues` | `toast`、`componentAction`、`custom` |
| `fieldsChange` | 字段变化 | `value` | `changedFields`、`allFields` | `toast`、`componentAction`、`custom` |

说明：

- P0 先保留 finish。
- Form 暴露方法：`submit`。
- Button 可以通过 `componentAction` 调用 Form.submit。
- valuesChange/fieldsChange 频率高，默认不允许 url。

### FormItem

| 事件名 | 文案 | 事件类别 | 事件数据 | 允许动作 |
| --- | --- | --- | --- | --- |
| 无 | 无 | 无 | 无 | 无 |

说明：

- 当前 FormItem 是 Form 的结构配置节点，不单独运行。
- FormItem 不建议出现在事件面板。

### Table

| 事件名 | 文案 | 事件类别 | 事件数据 | 允许动作 |
| --- | --- | --- | --- | --- |
| `rowClick` | 行点击 | `ui` | `record`、`index`、原始事件参数 | `toast`、`url`、`componentAction`、`custom` |
| `rowDoubleClick` | 行双击 | `ui` | `record`、`index`、原始事件参数 | `toast`、`url`、`componentAction`、`custom` |
| `rowSelect` | 行选中 | `value` | `record`、`selected`、`selectedRows` | `toast`、`componentAction`、`custom` |
| `selectionChange` | 选择变化 | `value` | `selectedRowKeys`、`selectedRows` | `toast`、`componentAction`、`custom` |
| `change` | 表格变化 | `value` | `pagination`、`filters`、`sorter`、`extra` | `toast`、`componentAction`、`custom` |
| `loadSuccess` | 加载成功 | `lifecycle` | `data` | `toast`、`componentAction`、`custom` |

## P0 推荐落地范围

先只落地这些事件，能覆盖当前物料区主要体验：

| 组件 | P0 事件 |
| --- | --- |
| Container | `click`、`doubleClick` |
| Card | `click`、`doubleClick` |
| Button | `click`、`doubleClick` |
| Text | `click`、`doubleClick` |
| Image | `click`、`doubleClick` |
| Divider | `click` |
| Alert | `click`、`close` |
| Input | `change`、`pressEnter`、`blur` |
| Select | `change` |
| Switch | `change` |
| Modal | `ok`、`cancel` |
| Form | `finish` |

P0 不做：

- Page lifecycle。
- Image load/error。
- Table rowClick/loadSuccess/loadError。
- Form finishFailed/valuesChange。
- FormItem/TableColumn 独立事件。

## P1 推荐增强范围

P0 稳定后，优先增强这些能力：

| 组件 | P1 事件 | 需要配套改造 |
| --- | --- | --- |
| Input | `focus`、`clear` | prod 透传 onFocus/onClear，事件数据提取 value |
| Select | `select`、`clear`、`focus`、`blur`、`openChange` | 支持 allowClear/showSearch/mode 等属性 |
| Switch | `click` | 事件数据提取 checked |
| Alert | `afterClose` | prod 使用 closable.afterClose 或包装兼容 |
| Image | `load`、`error` | dev/prod 包装 onLoad/onError |
| Modal | `afterClose`、`openChange` | 明确 Modal 内部 open 状态生命周期 |
| Form | `finishFailed`、`valuesChange` | 增加失败和变更事件数据 |
| Table | `rowClick`、`rowDoubleClick`、`change`、`loadSuccess`、`loadError` | 改造 Table prod 的数据加载与 onRow/onChange |

## P2 推荐增强范围

这些更适合数据源和变量系统成熟后再做：

| 组件 | P2 事件 | 原因 |
| --- | --- | --- |
| Page | `mount`、`unmount` | 需要统一生命周期触发器 |
| Container/Card | `mouseEnter`、`mouseLeave` | 需要处理大量 hover 事件和冒泡 |
| Select | `search`、`deselect` | 需要 showSearch 和多选模式 |

| Card | `tabChange` | 当前 Card 尚未支持 tabs |

## 动作过滤规则

动作弹窗不应该展示所有动作，而是按事件声明过滤。

### 默认规则

| 事件类别 | 默认动作 |
| --- | --- |
| `ui` | `toast`、`url`、`componentAction`、`custom` |
| `value` | `toast`、`componentAction`、`custom` |
| `submit` | `toast`、`url`、`componentAction`、`custom` |
| `overlay` | `toast`、`componentAction`、`custom` |
| `lifecycle` | `toast`、`componentAction`、`custom` |

### 覆盖规则

- `pressEnter` 可以允许 `url`，因为它常用于搜索/提交场景。
- `Divider.click` 不默认允许 `url`，因为 Divider 是装饰组件。
- `Modal.ok/cancel` 不默认允许 `url`，避免用户关闭弹窗时意外跳转。
- `Form.finish` 允许 `url`，因为提交成功后跳转很常见。
- `Table.rowClick` 允许 `url`，因为点击详情行跳转很常见。
- 高频事件如 `change`、`valuesChange`、`search` 默认不允许 `url`。

## 事件数据标准化

运行态应该把不同组件事件参数转成稳定数据，供动作引用。

| 事件 | 标准数据 |
| --- | --- |
| Button.click | `{ args, nativeEvent }` |
| Input.change | `{ value, args, nativeEvent }` |
| Input.pressEnter | `{ value, args, nativeEvent }` |
| Select.change | `{ value, option, args }` |
| Select.openChange | `{ open, args }` |
| Switch.change | `{ checked, args, nativeEvent }` |
| Form.finish | `{ values, args }` |
| Form.finishFailed | `{ values, errorFields, outOfDate, args }` |
| Modal.openChange | `{ open, args }` |
| Table.rowClick | `{ record, index, args, nativeEvent }` |
| Table.change | `{ pagination, filters, sorter, extra, args }` |
| Image.error | `{ src, args, nativeEvent }` |

P0 先保证：

```text
value / checked / values / args
```

P1 再补齐：

```text
record / index / option / pagination / filters / sorter / error
```

## 右侧事件面板交互规划

事件面板应该改成三段式：

```text
事件列表
  ↓
选择某个事件
  ↓
只展示该事件允许添加的动作
```

### 事件卡片

每个事件卡片展示：

- 事件名称，例如“点击事件”。
- 事件说明，例如“用户点击按钮时触发”。
- 已配置动作数量。
- 添加动作按钮。

### 添加动作弹窗

打开动作弹窗时，根据当前事件过滤动作：

```ts
allowedActions = event.allowedActions
```

例如：

- Input.change：不能选择 `url`。
- Button.click：可以选择 `url`。
- Form.finish：可以选择 `url`。
- Modal.cancel：默认不显示 `url`。

### 动作卡片

动作列表应该展示：

- 动作类型。
- 核心配置摘要。
- 编辑按钮。
- 删除按钮。
- 上移/下移按钮，动作按顺序执行。

## 实现顺序建议

3. 改造 `ActionModal`，根据 `allowedActions` 过滤动作 tab。
4. 改造 `ComponentEvent`，展示事件说明和动作数量。
5. 改造 `createEventData`，分别处理 Input、Select、Switch、Form 的事件数据。
