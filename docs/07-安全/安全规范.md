# 安全规范

## 鉴权

- 注册和登录之外的业务接口默认需要登录。
- 使用 JWT access token 保护接口。
- JWT secret 必须来自环境变量，不能写死在代码中。
- 前端请求时通过统一 axios 实例附加 token。

## 密码

- 密码不能明文保存。
- 使用 bcrypt 或 argon2 保存密码哈希。
- 登录失败不要提示“邮箱存在但密码错误”这类可枚举信息，统一提示账号或密码错误。

## 权限

- 项目和页面数据必须按当前用户隔离。
- 用户只能访问自己拥有的项目。
- 页面接口必须校验页面所属项目是否属于当前用户。
- 删除项目或页面需要后端二次权限校验，不能只依赖前端隐藏按钮。

## 输入校验

- 后端 DTO 必须校验用户输入。
- 字符串长度要有限制，例如项目名、页面名、路径。
- 页面 schema 必须是对象结构，不能接受任意字符串作为可执行代码。
- 文件上传必须限制类型、大小和存储位置。当前后端只允许图片、PDF、文本类白名单，并会校验最终路径不能逃逸 `UPLOAD_DIR`。

## HTTP 动作

- HTTP 动作由用户浏览器发起请求，不能把后端密钥、数据库连接串或服务端 token 注入动作配置。
- 开发环境默认不限制请求域名，方便联调。
- 生产环境建议配置 `VITE_LOWCODE_HTTP_ALLOWED_ORIGINS`，用逗号分隔允许访问的 origin，例如：

```text
VITE_LOWCODE_HTTP_ALLOWED_ORIGINS=https://api.example.com,https://crm.example.com
```

- 配置白名单后，HTTP 动作运行态会在 URL 归一化后校验 origin，不在白名单内的请求会被阻止并记录运行日志。

## 自定义 JS 风险

当前编辑器支持 `customJS`，预览时会通过 `new Function` 执行用户配置代码。该能力有明显安全风险。

第一阶段要求：

- 不扩大 `customJS` 能力。
- 不在未登录公开页面中允许不可信用户编辑 customJS。
- 上线前需要考虑禁用、沙箱化或仅允许可信管理员使用。
- 不把服务端 token、密钥或敏感上下文暴露给 customJS。

## 公开发布页

- 公开访问使用 `publicId`，不要把自增 `pageId` 作为公开 URL。
- 公开接口不需要 JWT，但只能返回页面名称、路径、schema、发布时间等公开字段。
- 公开页面读取 `publishedVersionId` 指向的发布快照，不直接读取可编辑的 `Page.schema` 草稿。
- 公开运行态必须传 `allowCustomJS={false}`，不能执行用户配置的 `customJS`。
- 取消发布后，公开接口应返回 404，不继续暴露历史快照。

## 环境变量

- `.env` 不进入 git。
- 提交 `.env.example` 说明必需配置。
- 数据库连接串、JWT secret、对象存储密钥只放环境变量。
- 生产环境 `JWT_SECRET` 必须足够长，不能使用示例弱密钥。

## API 返回

- 不返回 passwordHash。
- 不返回内部异常堆栈。
- 不把数据库连接错误、SQL 或 Prisma 原始错误直接返回给用户。

## 部署前安全检查

- 确认生产 JWT secret 足够随机。
- 确认数据库不暴露公网或有强密码。
- 确认 CORS 只允许可信前端域名。
- 确认 `VITE_LOWCODE_HTTP_ALLOWED_ORIGINS` 已按业务 API 域名收口。
- 确认 `UPLOAD_DIR` 指向专用目录，且上传文件通过 Nginx 或 API 以受控路径访问。
- 确认日志中没有 token、密码、连接串。
- 确认发布页不会执行非可信用户注入的危险脚本。
