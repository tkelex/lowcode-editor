# 为什么低代码编辑器需要后端和数据库

这份文档用于解释：为什么当前低代码编辑器项目不能只停留在前端，为什么需要 NestJS 后端和 PostgreSQL 数据库，以及它们在项目中的主要作用。

## 1. 一句话总结

后端和数据库的作用是把项目从“只能在浏览器里临时编辑的前端 demo”，升级成“可以登录、保存、恢复、回滚和发布的产品雏形”。

如果没有后端和数据库，项目大致只能做到：

```text
打开页面
  ↓
拖拽组件
  ↓
组件存在浏览器内存 / localStorage
  ↓
刷新、换设备或清缓存后容易丢失
```

有了后端和数据库后，项目可以做到：

```text
用户注册 / 登录
  ↓
创建项目
  ↓
创建页面
  ↓
编辑页面
  ↓
保存页面 schema 到数据库
  ↓
再次打开页面时恢复组件树
  ↓
查看历史版本
  ↓
回滚旧版本
  ↓
发布成公开页面
```

## 2. 数据库的主要作用

数据库负责长期保存项目数据。

当前核心数据模型包括：

```text
User          用户
Project       项目
Page          页面当前草稿
PageVersion   页面历史版本 / 发布快照
```

其中最重要的是：

```prisma
Page.schema
PageVersion.schema
```

它们保存的是低代码页面的组件树。

例如画布里有：

```text
Page
  ├─ Container
  │   ├─ Text
  │   └─ Button
  └─ Modal
```

前端会把它表示成类似这样的 JSON schema：

```ts
{
  schemaVersion: '1.0.0',
  components: [
    {
      id: 1,
      name: 'Page',
      children: [
        {
          id: 2,
          name: 'Container',
          children: [
            { id: 3, name: 'Text', props: { text: '标题' } },
            { id: 4, name: 'Button', props: { text: '提交' } }
          ]
        }
      ]
    }
  ]
}
```

数据库的作用就是把这棵组件树保存下来。

再次打开页面时：

```text
数据库 Page.schema
  ↓
后端返回页面数据
  ↓
前端 setComponents
  ↓
画布恢复成上次编辑的样子
```

所以数据库解决的是“页面能不能长期保存、再次恢复”的问题。

## 3. 后端的主要作用

后端负责业务规则和安全控制。

可以这样理解：

```text
前端：负责用户操作和界面
后端：负责判断这个操作能不能做、应该怎么做
数据库：负责把结果长期存下来
```

## 4. 后端在当前项目里的具体职责

### 4.1 登录注册

注册和登录由后端处理：

```text
POST /api/auth/register
POST /api/auth/login
```

后端负责：

- 保存用户账号。
- 密码加密存储。
- 登录时校验密码。
- 登录成功后签发 JWT token。

如果没有后端，前端无法安全地管理真实用户账号。

### 4.2 项目和页面管理

用户可以创建项目和页面：

```text
POST /api/projects
POST /api/projects/:projectId/pages
```

后端负责：

- 创建项目。
- 创建页面。
- 记录页面属于哪个项目。
- 记录项目属于哪个用户。

例如：

```text
用户 A 的项目
  ├─ 页面 1
  └─ 页面 2

用户 B 的项目
  └─ 页面 3
```

后端必须保证：

```text
用户 A 不能访问或修改用户 B 的页面
```

这不能只靠前端隐藏按钮，因为用户可以绕过界面直接请求接口。

### 4.3 保存页面 schema

用户点击保存时：

```text
前端 Zustand components
  ↓
PATCH /api/pages/:id
  ↓
后端校验页面归属
  ↓
写入数据库 Page.schema
```

后端不是简单转发数据，而是要先判断：

```text
当前登录用户是不是这个页面的 owner？
```

只有是自己的页面，才允许保存。

### 4.4 版本历史和回滚

当前保存版本的流程是：

```text
每次保存
  ↓
更新 Page.schema
  ↓
创建 PageVersion
```

回滚时：

```text
选择历史版本
  ↓
POST /api/pages/:id/rollback
  ↓
后端校验页面 owner
  ↓
确认 version 属于这个 page
  ↓
把 PageVersion.schema 恢复到 Page.schema
```

后端在这里很重要，因为它要防止：

```text
用户拿别人的 versionId 去回滚页面
用户拿自己的 token 操作别人的页面
```

所以版本管理不能只做在前端。

### 4.5 页面发布

发布功能也需要后端。

当前发布流程是：

```text
点击发布
  ↓
前端先保存当前画布
  ↓
POST /api/pages/:id/publish
  ↓
后端创建 PageVersion(source = publish)
  ↓
更新 Page.publishedVersionId
  ↓
生成公开访问地址
```

公开页面访问：

```text
GET /api/public/pages/:publicId
```

后端负责：

- 判断页面是否已经发布。
- 根据 `publicId` 找页面。
- 读取发布快照。
- 只返回公开页面需要的数据。
- 不返回用户信息、项目 owner 信息等内部字段。

所以后端是发布能力的核心控制层。

## 5. 为什么不能只用 localStorage

`localStorage` 只能解决很小的 demo 问题。

它的问题包括：

```text
只能存在当前浏览器
换电脑就没了
清缓存就没了
无法多用户隔离
无法安全登录
无法多人协作
无法可靠发布
无法被后端校验权限
无法做真实部署
```

如果页面数据只存在浏览器里，那么：

```text
换到另一个浏览器
换到另一台电脑
清理浏览器缓存
重新部署前端
```

都可能导致数据丢失。

而数据库可以长期保存数据：

```text
用户登录
  ↓
从数据库读取自己的项目和页面
  ↓
继续编辑
```

## 6. 为什么不能让前端直接连接数据库

前端不能直接连接数据库，因为这样非常不安全。

如果前端直接连接数据库，就必须把数据库地址、账号和密码暴露给浏览器：

```text
DATABASE_URL=postgresql://user:password@host:5432/db
```

任何人都可以在浏览器里看到这些信息，从而拿到数据库权限。

正确架构应该是：

```text
浏览器前端
  ↓ HTTP API
NestJS 后端
  ↓ Prisma
PostgreSQL 数据库
```

前端不能直接操作数据库，必须通过后端接口。

## 7. 当前项目里前端、后端、数据库的分工

```text
React 前端
  负责：编辑器界面、拖拽、属性配置、预览、点击保存和发布按钮

NestJS 后端
  负责：登录鉴权、权限校验、业务规则、API、版本回滚、发布逻辑

PostgreSQL 数据库
  负责：长期保存用户、项目、页面、schema、历史版本、发布快照
```

保存时：

```text
React Zustand components
  ↓
NestJS PATCH /api/pages/:id
  ↓
校验 owner
  ↓
PostgreSQL Page.schema
```

发布时：

```text
React 点击发布
  ↓
NestJS POST /api/pages/:id/publish
  ↓
PostgreSQL 创建 PageVersion(source = publish)
  ↓
访客访问 /publish/:publicId
```

## 8. 面试回答版本

可以这样回答：

> 这个项目最开始如果只有前端，它更像一个低代码编辑器 demo，组件树只存在浏览器状态里，刷新或换设备后就不可靠。引入后端和数据库后，我可以把用户、项目、页面和页面 schema 持久化下来。前端负责拖拽、配置和预览，后端负责登录鉴权、权限校验、保存、版本回滚和发布逻辑，数据库负责长期保存 Page.schema 和 PageVersion.schema。这样项目就从“能拖拽的页面”变成了“能保存、能恢复、能回滚、能发布”的低代码平台雏形。

如果需要更短，可以说：

> 后端负责业务规则和安全，数据库负责长期保存数据。没有它们，这个低代码编辑器只能是本地 demo；有了它们，页面才能按用户保存、恢复、管理版本并发布给访客访问。
