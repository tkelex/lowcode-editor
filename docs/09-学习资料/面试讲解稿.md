# 低代码编辑器项目面试讲解稿

本文是面试表达提纲，不作为工程事实来源。最新实现以源码和正式文档为准。

## 一句话介绍

这是一个基于 React + TypeScript + NestJS + PostgreSQL 的低代码页面编辑器。用户可以登录后创建项目和页面，通过拖拽组件、配置属性/样式/事件生成页面 schema，保存到后端数据库，并支持版本历史、回滚和公开发布访问。

## 项目解决的问题

原始编辑器 demo 通常只能在前端维护临时状态，刷新后数据丢失。本项目把它扩展成真实产品闭环：

```text
登录 -> 项目 -> 页面 -> 编辑器 -> 保存 schema
  -> 版本历史 -> 回滚 -> 发布 -> 公开访问
```

## 技术栈

| 层 | 技术 |
| --- | --- |
| 前端 | Vite、React、TypeScript、Ant Design、Zustand、React DnD、Monaco |
| 后端 | NestJS、TypeScript、JWT、Passport |
| 数据库 | PostgreSQL、Prisma |
| 工程 | Docker Compose、lint、build、node test、Playwright、API smoke |

## 前端怎么设计

可以这样讲：

> 编辑器的本质是维护一棵组件树。拖拽组件就是往树里插入节点，属性面板修改节点的 props 和 styles，事件面板修改 props.onEvent。保存时把组件树包装成页面 schema 发给后端。

关键模块：

- `src/app/App.tsx`：登录、项目面板、编辑器、公开页的视图分发。
- `src/editor/stores/components.tsx`：组件树和撤销重做。
- `src/editor/registry/component-config.tsx`：物料注册表。
- `src/editor/components/EditArea`：编辑态渲染。
- `src/editor/runtime/Preview`：预览和公开页运行态。

## 编辑态和预览态

```text
编辑态：dev 物料 + 选择/拖拽/设置能力
预览态：prod 物料 + 事件动作执行
公开页：prod 物料 + 发布快照 + 禁用 customJS
```

## 后端怎么设计

可以这样讲：

> 后端负责用户、项目、权限、页面持久化、版本和发布。页面 schema 存在 PostgreSQL 的 JSONB 字段里，保存和发布前都会经过共享 schema 包的迁移和校验。

关键模块：

- `auth`：注册、登录、JWT。
- `projects`：项目和成员权限。
- `pages`：页面 CRUD、版本、发布。
- `admin`：平台治理。
- `packages/lowcode-schema`：前后端共享 schema 规则。

## 保存、版本、发布区别

| 概念 | 含义 |
| --- | --- |
| `Page.schema` | 当前编辑草稿 |
| `PageVersion.schema` | 保存、回滚或发布时生成的快照 |
| 发布快照 | `source=publish` 的 PageVersion |
| 公开页 | 读取 `publishedVersionId` 指向的快照 |

发布后继续编辑草稿，不会自动影响公开页，直到再次发布。

## 权限和安全

- JWT 用于登录鉴权。
- 项目角色分为 owner/editor/viewer。
- 后端强制校验权限，前端隐藏按钮只是体验优化。
- 管理员接口由 `AdminGuard` 从数据库复查 role/status。
- 公开发布页必须禁用 customJS。
- 密码 hash 保存，不返回 passwordHash。

## 可以重点讲的亮点

- 从纯前端 demo 扩展到全栈闭环。
- 用共享 schema 包保证前后端校验一致。
- 用 PageVersion 支持保存历史、回滚和发布快照。
- 通过 dev/prod 物料拆分编辑态和运行态。
- 事件系统支持 toast、跳转、HTTP、变量、组件控制和条件动作。
- 引入 API smoke、schema test 和编辑器 e2e 保护核心链路。

## 常见追问回答

### 为什么 schema 存 JSONB？

页面结构变化快，组件 props 差异大，用 JSONB 能灵活保存组件树，同时仍可挂在 Page、Project、User 等关系模型下管理。

### 为什么需要后端？

后端提供登录、权限、持久化、版本、发布和审计。只用 localStorage 无法支持多用户、跨设备、版本和公开访问。

### 为什么要共享 schema 包？

保存、发布、回滚、公开页都依赖同一套 schema 规则。共享包能避免前端放行、后端拒绝，或旧数据无法恢复的问题。

### 项目最大风险是什么？

组件树 store、物料 registry、Preview 运行态和 schema migration 影响面最大。修改时要小步验证，避免修一个物料破坏保存或发布链路。
