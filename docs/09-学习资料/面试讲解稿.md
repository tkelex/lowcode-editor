# 低代码编辑器项目面试讲解稿

这份文档用于面试时快速介绍本项目。目标不是逐行解释代码，而是让面试官听清楚：项目做什么、为什么这样设计、我负责了什么、遇到什么问题、学到了什么。

## 1. 一句话介绍项目

这是一个基于 React + TypeScript 的低代码编辑器项目，已经从前端 demo 扩展成一个具备登录、项目管理、页面管理、页面 schema 保存、历史版本、回滚、版本记录删除和页面发布访问能力的全栈产品雏形。

可以这样说：

> 这个项目是一个低代码页面编辑器，用户可以登录后创建项目和页面，在编辑器里拖拽组件、配置属性和事件，然后把页面结构保存到后端数据库。后续再次打开页面时，可以从数据库恢复组件树。同时我还做了页面历史版本、回滚、删除历史版本记录和页面发布访问，让它更接近真实产品。

## 2. 项目解决的问题

原始低代码编辑器通常只停留在前端 demo：

```text
打开页面 → 拖拽组件 → Zustand 本地状态 → 刷新后丢失
```

这个项目要解决的是：让编辑器具备真实产品闭环。

核心闭环是：

```text
用户注册 / 登录
  ↓
创建项目
  ↓
创建页面
  ↓
打开编辑器
  ↓
拖拽组件、配置属性、样式、事件
  ↓
保存页面 schema 到 PostgreSQL
  ↓
再次打开页面，从后端恢复组件树
  ↓
查看历史版本、回滚旧版本、删除无用版本记录
  ↓
发布页面，访客无需登录访问公开地址
```

所以它不是单纯展示 UI，而是在做一个“能保存、能恢复、能管理历史版本、能发布访问”的低代码平台基础能力。

## 4. 前端核心设计

前端编辑器主要分为三块：

```text
左侧：物料区 / 大纲 / 源码
中间：编辑画布
右侧：属性设置面板
顶部：保存、发布、版本历史、预览切换
```

核心状态放在 Zustand 里：

```ts
components       // 当前页面组件树
curComponentId   // 当前选中的组件 ID
curComponent     // 当前选中的组件
mode             // edit 或 preview
```

组件树结构大致是：

```ts
{
  id: number;
  name: string;
  props: any;
  styles?: CSSProperties;
  desc: string;
  children?: Component[];
  parentId?: number;
}
```

面试时可以这样讲：

> 编辑器的本质是维护一棵组件树。拖拽组件就是往这棵树里插入节点，属性面板修改的也是节点上的 props 或 styles。保存时，我把这棵组件树包装成页面 schema，提交给后端保存。

## 5. 编辑态和预览态

项目里有两套渲染：

```text
dev 组件：编辑态使用，支持选中、hover、拖拽、辅助边框
prod 组件：预览态使用，更接近真实页面运行效果
```

编辑态：

```text
组件树 → componentConfig[name].dev → 编辑画布
```

预览态：

```text
组件树 → componentConfig[name].prod → 运行态预览
```

可以这样说：

> 低代码编辑器不能只用一套组件。编辑态组件需要服务编辑体验，比如拖拽和选中；预览态组件要尽量接近真实运行页面。所以项目里每个物料通常都有 dev 和 prod 两个实现。

## 6. 后端核心设计

后端采用 NestJS 常见分层：

```text
Controller  接收 HTTP 请求
Service     处理业务逻辑
DTO         校验请求体
Guard       控制鉴权
Prisma      访问数据库
```

当前核心模块：

```text
AuthModule      注册、登录、获取当前用户
ProjectsModule  项目 CRUD
PagesModule     页面 CRUD、schema 保存、版本历史、回滚、版本删除、发布和公开读取
PrismaModule    数据库访问
```

可以这样说：

> 我把后端按业务模块拆分，Controller 只负责路由入口，真正的权限校验和业务逻辑放在 Service。所有业务接口都走 JWT Guard，并且项目和页面操作都要校验 owner，避免跨用户访问。

## 7. 数据库模型

当前核心模型：

```text
User
Project
Page
PageVersion
```

关系大致是：

```text
User 1 - N Project
Project 1 - N Page
Page 1 - N PageVersion
```

最核心字段是：

```prisma
Page.schema Json
PageVersion.schema Json
```

解释方式：

> 低代码页面的结构非常灵活，不同页面可能有不同组件、属性、样式和事件。如果全部拆成关系表会很复杂，所以当前阶段我把页面组件树作为 JSON schema 保存到 PostgreSQL 的 JSONB 字段里。`Page.schema` 表示当前页面状态，`PageVersion.schema` 表示历史快照。

## 8. 页面保存流程

保存流程：

```text
用户点击保存
  ↓
前端读取 Zustand 里的 components
  ↓
包装成 PageSchema
  ↓
PATCH /api/pages/:id
  ↓
后端校验页面归属
  ↓
更新 Page.schema
  ↓
创建 PageVersion 历史版本
```

可以这样说：

> 保存不是简单地存在 localStorage，而是把当前组件树提交给后端。后端会先校验这个页面是不是当前用户的，然后更新当前页面 schema，并生成一条历史版本记录。

## 9. 页面回滚流程

回滚流程：

```text
打开版本历史
  ↓
GET /api/pages/:id/versions
  ↓
选择某个版本
  ↓
POST /api/pages/:id/rollback
  ↓
后端把 PageVersion.schema 恢复到 Page.schema
  ↓
创建一条 source = rollback 的新版本
  ↓
前端 setComponents 更新画布
```

可以这样说：

> 回滚不是删除后面的历史，而是把某个历史快照复制回当前页面，并且再生成一条新的 rollback 版本。这样历史记录是完整的，可以知道用户什么时候做过回滚。

## 10. 删除历史版本记录

删除版本记录流程：

```text
打开版本历史
  ↓
点击某条版本的删除
  ↓
Popconfirm 二次确认
  ↓
DELETE /api/pages/:id/versions/:versionId
  ↓
后端校验页面 owner 和版本归属
  ↓
删除 PageVersion
  ↓
前端刷新版本列表
```

重点说明：

> 删除历史版本只删除 `PageVersion` 快照，不修改 `Page.schema`，所以不会影响当前编辑器画布。删除后该版本不能再用于回滚。

后端为了安全，删除时不是只按 versionId 删除，而是同时匹配：

```ts
deleteMany({ where: { id: versionId, pageId } })
```

这样可以避免跨页面误删。

## 11. 页面发布流程

发布流程：

```text
编辑器点击发布
  ↓
POST /api/pages/:id/publish
  ↓
后端校验页面 owner
  ↓
基于当前 Page.schema 创建 PageVersion(source = publish)
  ↓
更新 Page.isPublished / publicId / publishedVersionId
  ↓
访客访问 /publish/:publicId
  ↓
GET /api/public/pages/:publicId
  ↓
公开页面渲染发布快照
```

可以这样说：

> 发布不是直接把当前草稿暴露出去，而是生成一次发布快照。公开页面读取 `publishedVersionId` 指向的 `PageVersion`，所以用户发布后继续编辑草稿，不会自动影响已经公开的页面，直到再次发布。

关键设计：

- 公开 URL 使用 `publicId`，不暴露递增 `pageId`。
- 公开接口不需要登录，但只返回公开页面需要的字段。
- 发布快照存放在 `PageVersion(source = publish)`，和历史版本复用同一套快照模型。
- 公开运行态禁用 `customJS`，避免访客页面执行用户配置的脚本。

## 12. 鉴权和权限控制

当前权限模型是 owner 模型：

```text
用户只能访问自己的项目
用户只能访问自己项目下的页面
页面版本接口也必须先校验页面归属
```

可以这样说：

> 我没有只在前端隐藏按钮，而是在后端 Service 层做 owner 校验。比如操作页面版本之前，先确认 page 属于当前用户，再确认 version 属于这个 page。

## 14. 项目中遇到的问题和解决方式

### 13.1 创建第二个页面报错

问题：同一个项目里创建第二个页面时报错。

原因：页面路径默认都是 `/home`，但数据库有唯一约束：

```prisma
@@unique([projectId, routePath])
```

解决：

- 前端创建页面时自动生成 `/page-1`、`/page-2` 这种不重复路径。
- 后端捕获 Prisma 唯一约束错误，返回更明确的 409 Conflict。

可以总结：

> 这个问题让我意识到，产品体验和数据库约束要配合。后端要保证规则正确，前端也要减少用户触发错误的概率。

### 13.2 预览模式看不到模块

问题：点击预览后，有些布局组件看起来像没渲染。

原因：预览态使用 prod 组件，空容器没有边框和最小高度，看起来就是空白。

解决：

- 给 Page、Container 的 prod 组件增加最小高度和视觉边界。
- 对缺少 prod 组件的物料显示明确占位，而不是直接返回 null。

可以总结：

> 低代码编辑器里，空容器也需要可视化提示，否则用户会误以为组件丢了。

### 13.3 Prisma Client 没更新导致构建失败

问题：新增 `PageVersion` 后，后端构建提示 `pageVersion` 不存在。

原因：修改了 `schema.prisma`，但没有重新生成 Prisma Client。

解决：

```bash
npm run prisma:generate --prefix server
```

可以总结：

> Prisma 的类型来自生成的 Client，改完数据库模型后要重新 generate，否则 TypeScript 还不知道新模型。

### 13.4 Windows 下 Prisma DLL 被占用

问题：Prisma generate 时遇到 Windows DLL rename 权限错误。

原因：本地后端 dev server 或 Prisma 进程占用了 query engine 文件。

解决：停止占用进程后重新执行 generate。

## 17. 如果面试官问：为什么用 JSONB 保存页面 schema？

可以这样回答：

> 低代码页面的组件结构变化很大，不同组件有不同 props、styles、events。如果全部拆成关系表，早期实现成本很高，也不灵活。当前阶段更适合把整棵组件树作为 JSON 保存到 PostgreSQL 的 JSONB 字段。这样读写简单，也方便后续扩展。等将来有搜索、统计或发布优化需求时，再考虑把部分关键字段结构化。

## 18. 如果面试官问：回滚是怎么设计的？

可以这样回答：

> 每次保存页面时，后端除了更新 `Page.schema`，还会创建一条 `PageVersion`。回滚时，后端先校验页面属于当前用户，再确认版本属于这个页面，然后把目标版本的 schema 复制回 `Page.schema`。回滚本身也会生成一条新的 `PageVersion`，source 是 rollback，这样历史记录不会断。

## 19. 如果面试官问：发布是怎么设计的？

可以这样回答：

> 发布时后端不会直接让公开页面读取当前 `Page.schema`，而是先基于当前 schema 创建一条 `PageVersion(source = publish)` 快照，然后把 `Page.publishedVersionId` 指向这条快照。公开 URL 用不可枚举的 `publicId`，访客访问 `/publish/:publicId` 时，公开接口只返回页面名称、路径、schema 和发布时间，不返回用户或项目信息。这样可以做到草稿和线上页面隔离，用户继续编辑草稿不会影响已发布页面。

## 20. 如果面试官问：怎么保证用户不能操作别人的页面？

可以这样回答：

> 首先业务接口都使用 JWT Guard，必须登录。其次在 Service 层查询页面时，会通过页面所属项目校验 owner。版本相关操作也不是只查 versionId，而是先确认 page 属于当前用户，再确认 version 属于这个 page。这样可以避免跨用户或跨页面访问。

## 21. 如果面试官问：这个项目还有哪些可以优化？

可以从这些方向回答：

1. 发布能力增强：增加取消发布入口、发布记录、SEO、自定义域名和访问统计。
2. 权限协作：支持项目成员和 owner/editor/viewer 角色。
3. 自定义 JS 沙箱：当前编辑器预览支持 customJS，生产环境需要沙箱或更严格限制。
4. CI/CD：增加自动 lint、build、test。
5. 部署能力：补充 Dockerfile、Nginx、生产环境变量和 migration 流程。
6. 性能优化：大型组件树下减少不必要渲染，优化 Zustand selector 和组件递归渲染。
7. 版本管理增强：支持版本备注、版本对比、按时间筛选。

可以这样总结：

> 当前项目已经完成低代码平台最小闭环，下一步我会优先做权限协作、发布增强和部署上线，因为这些能力决定它能不能从编辑器进一步变成真实可用的平台。

> 为了让它更像真实产品，我又做了版本管理和发布能力。每次保存不仅更新当前页面，还会生成一条 `PageVersion` 历史快照。用户可以在版本历史里选择旧版本回滚，回滚时后端把历史 schema 恢复到当前页面，并再生成一条 rollback 来源的新版本。删除历史版本记录只会删除 `PageVersion`，不会影响当前 `Page.schema`。发布时，后端会基于当前页面 schema 创建一条 publish 来源的快照，并把 `publishedVersionId` 指向它。访客访问 `/publish/:publicId` 时看到的是发布快照，不需要登录，也不会被后续草稿编辑影响。
>
> 这个项目里我比较重视权限和安全边界，项目、页面、版本接口都要先校验 owner，防止跨用户访问。公开发布页虽然不需要登录，但只返回公开字段，并且禁用 `customJS`，避免访客页面执行用户配置脚本。过程中也解决过一些真实问题，比如创建第二个页面路径重复、预览模式空容器不可见、Prisma Client 没更新导致构建失败等。整体收获是我不只是做了前端交互，还完整理解了一个产品从编辑、保存、恢复、版本管理到发布访问的闭环。
