# 功能演示流程 Feature Walkthrough

这份文档用于演示项目核心功能，可以给面试官、协作者或未来自己快速说明：项目怎么跑、用户怎么使用、每一步背后发生了什么。

## 1. 演示目标

通过一条完整路径展示低代码编辑器的核心闭环：

```text
注册 / 登录
  ↓
创建项目
  ↓
创建页面
  ↓
进入编辑器
  ↓
拖拽和配置组件
  ↓
保存页面
  ↓
刷新后恢复页面
  ↓
查看版本历史
  ↓
回滚旧版本
  ↓
删除无用历史版本
  ↓
发布页面
  ↓
访客无需登录访问公开页面
```

演示重点不是组件数量，而是说明这个项目已经具备“编辑器产品”的基础能力：可保存、可恢复、可回滚、可管理版本、可发布访问，并且物料区已经从扁平 demo 列表升级为分类搜索面板。

## 2. 演示前准备

### 2.1 启动数据库

```bash

```



## 3. 演示流程

## 3.1 注册或登录

操作：

1. 打开前端页面。
2. 如果没有账号，先注册。
3. 注册成功后进入项目面板。
4. 已有账号可以直接登录。

讲解重点：

> 项目不是纯前端 demo，而是有真实用户体系。登录成功后，前端会保存后端返回的 access token，后续业务接口会带上 JWT。

背后发生的事：

```text
POST /api/auth/register
或
POST /api/auth/login
  ↓
后端校验用户信息
  ↓
返回 accessToken
  ↓
前端保存 token
  ↓
后续请求自动带 Authorization header
```

相关文件：

```text
src/features/auth/AuthView.tsx
src/api/auth.ts
src/api/http.ts
server/src/modules/auth/auth.controller.ts
server/src/modules/auth/auth.service.ts
```

## 3.2 创建项目

操作：

1. 在项目面板点击“新建项目”。
2. 输入项目名称和描述。
3. 创建成功后，项目出现在列表里。

讲解重点：

> 项目是页面的上层容器。后续如果做团队协作、角色权限或发布管理，都会以项目为边界。

背后发生的事：

```text
POST /api/projects
  ↓
JwtAuthGuard 校验登录态
  ↓
CurrentUser 获取当前用户
  ↓
ProjectsService 创建 ownerId = 当前用户的项目
```

相关文件：

```text

```

## 3.3 创建页面

操作：

1. 进入某个项目。
2. 点击“新建页面”。
3. 输入页面名称和路径。
4. 创建成功后页面出现在当前项目下。

讲解重点：

> 页面是低代码编辑器真正编辑的对象。每个页面有自己的 `routePath` 和 `schema`。

背后发生的事：

```text
POST /api/projects/:projectId/pages
  ↓
后端确认 project 属于当前用户
  ↓
创建 Page
  ↓
如果前端没有传 schema，后端写入默认 Page schema
```

默认 schema 类似：

```text
Page
```

也就是组件树的根节点。

相关文件：

```text

server/src/modules/pages/pages.service.ts

```

## 3.4 进入编辑器

操作：

1. 在页面列表中点击某个页面。
2. 进入低代码编辑器。
3. 可以看到左侧物料区、中间画布、右侧设置面板、顶部操作栏。

讲解重点：

> 打开页面时，前端不是创建一份空状态，而是先向后端读取页面 schema，再写入 Zustand，恢复页面组件树。

背后发生的事：

```text
GET /api/pages/:id
  ↓
后端校验页面所属项目 owner
  ↓
返回 Page.schema
  ↓
前端 setComponents(page.schema.components)
  ↓
进入编辑器
```

相关文件：

```text

```

## 3.5 拖拽组件和配置属性

操作：

1. 从左侧物料区搜索或按分类找到组件。
2. 拖拽 Text、Image、Card、Alert、Input、Select、Switch 等组件到画布。
3. 可以把 Text、Button、Input 等组件拖入 Container 或 Card。
4. 点击画布中的组件。
5. 在右侧设置面板修改属性、样式或事件。
6. 切到「事件」面板，给 Text/Button/Input/Switch 等组件绑定“消息提示”或“访问链接”动作。

讲解重点：

> 编辑器核心就是维护一棵组件树。拖拽相当于新增节点，属性面板相当于修改节点的 props 或 styles，事件面板会把动作配置保存到 `component.props.onEvent[eventName].actions`，预览和发布运行态再把这些事件注入到对应 prod 组件。

组件节点结构：

```ts
{
  id: number;
  name: string;
  props: any;
  styles?: CSSProperties;
  desc: string;
  children?: Component[];
  parentId?: number;
}
```

背后发生的事：

```text
拖拽组件
  ↓
addComponent 写入 Zustand components
  ↓
点击组件
  ↓
setCurComponentId 记录当前组件
  ↓
设置面板修改 props/styles/events
  ↓
更新组件树
```

相关文件：


## 3.7 保存页面

操作：

1. 在编辑器中完成组件配置。
2. 点击顶部“保存”。
3. 等待提示“页面已保存，并生成历史版本”。

讲解重点：

> 保存时，前端会把当前组件树包装成 PageSchema，提交给后端。后端会更新当前页面，并生成一条历史版本。

背后发生的事：

```text
点击保存
  ↓
读取 Zustand components
  ↓
包装成 PageSchema
  ↓
PATCH /api/pages/:id
  ↓
后端校验页面 owner
  ↓
更新 Page.schema
  ↓
创建 PageVersion(source = save)
```

PageSchema 大致结构：

```ts
{
  schemaVersion: '1.0.0',
  pageId,
  components,
  metadata: {
    updatedAt: string
  }
}
```

相关文件：

```text
src/editor/components/Header/index.tsx

server/src/modules/pages/pages.controller.ts
server/src/modules/pages/pages.service.ts
server/prisma/schema.prisma
```

讲解重点：

## 3.10 回滚历史版本

操作：

1. 在版本历史中选择某个旧版本。
2. 点击“回滚”。
3. 在确认框中确认。
4. 画布恢复为旧版本内容。
5. 版本列表新增一条 rollback 来源的新版本。

讲解重点：

> 回滚不是删除历史记录，而是把旧快照复制回当前页面，并生成一条新的 rollback 版本。这样历史记录更完整。

背后发生的事：

```text
POST /api/pages/:id/rollback
  ↓
校验页面 owner
  ↓
确认 version 属于该 page
  ↓
把 PageVersion.schema 写回 Page.schema
  ↓
创建 PageVersion(source = rollback)
  ↓
前端 setComponents 更新画布
```

相关文件：

```text
server/src/modules/pages/dto/rollback-page.dto.ts
server/src/modules/pages/pages.controller.ts
server/src/modules/pages/pages.service.ts
src/editor/components/Header/index.tsx
```

## 3.11 删除历史版本记录

操作：

1. 打开版本历史。
2. 点击某个版本的“删除”。
3. 在确认框中确认删除。
4. 版本列表刷新，该版本消失。
5. 当前画布内容不变化。

讲解重点：

> 删除历史版本只删除 PageVersion 快照，不会修改当前 Page.schema，所以不会影响当前页面内容。

背后发生的事：

```text
DELETE /api/pages/:id/versions/:versionId
  ↓
校验页面 owner
  ↓
同时匹配 pageId 和 versionId
  ↓
删除 PageVersion
  ↓
前端刷新版本列表
```

后端删除使用：

```ts
deleteMany({ where: { id: versionId, pageId } })
```

这样可以避免跨页面误删。

相关文件：

```text
server/src/modules/pages/pages.controller.ts
server/src/modules/pages/pages.service.ts
src/api/pages.ts
src/editor/components/Header/index.tsx
```

## 3.12 发布页面并公开访问

操作：

1. 在编辑器中完成组件配置。
2. 点击顶部“发布”。
3. 复制或记录提示中的 `/publish/:publicId` 公开地址。
4. 在无痕窗口或未登录状态打开公开地址。
5. 确认页面不需要登录，且只展示运行态页面。

讲解重点：

> 发布按钮会先把当前画布组件树保存到 `Page.schema`，再基于数据库里的 `Page.schema` 生成一条 `PageVersion(source = publish)` 快照。公开页面读取的是这次发布快照，所以后续继续编辑草稿不会立刻影响公开页面。

背后发生的事：

```text
点击发布
  ↓
前端先保存当前画布 components 到 Page.schema
  ↓
POST /api/pages/:id/publish
  ↓
后端校验页面 owner
  ↓
基于当前 Page.schema 创建 PageVersion(source = publish)
  ↓
更新 Page.isPublished / publicId / publishedVersionId
  ↓
访客访问 /publish/:publicId
  ↓
GET /api/public/pages/:publicId
  ↓
读取 publishedVersionId 指向的快照
  ↓
公开页面用 Preview 只读渲染
```

安全点：

- 公开地址使用不可枚举的 `publicId`，不直接暴露递增 `pageId`。
- 公开接口不需要 JWT，但只返回公开页面需要的字段。
- 公开运行态传 `allowCustomJS={false}`，不会执行用户配置的 `customJS`。
- 如果页面取消发布，公开接口会返回 404。

相关文件：

```text

src/editor/components/Header/index.tsx
src/editor/components/Preivew/index.tsx
server/src/modules/pages/pages.controller.ts
server/src/modules/pages/pages.service.ts

```

## 4. 演示时可以强调的技术点

### 4.1 组件树是编辑器核心

低代码编辑器不是直接保存 HTML，而是保存抽象的组件树：

```text
Page
  ├─ Container
  │   ├─ Button
  │   └─ Text
  └─ Modal
```

这样后续可以继续编辑、预览、发布和版本管理。

### 4.2 Page.schema 和 PageVersion.schema 分工明确

```text
Page.schema        当前页面草稿
PageVersion.schema 历史版本或发布快照
```

保存会更新当前页面并生成历史版本；回滚会把历史版本恢复为当前页面；删除历史版本不会影响当前页面；发布会创建 `source = publish` 的快照供公开页面读取。

### 4.3 权限校验放在后端

项目、页面和版本相关操作都不能只靠前端控制。后端会校验：

```text
当前用户是否拥有该项目
当前页面是否属于该项目
当前版本是否属于该页面
```

### 4.4 编辑态和预览态分离

```text
dev  组件：服务编辑器交互
prod 组件：服务预览 / 运行效果
```

### 4.5 发布快照和草稿隔离

公开页面读取 `publishedVersionId` 指向的 `PageVersion`，不是直接读取当前可编辑的 `Page.schema`。

```text
Page.schema                 编辑器草稿
PageVersion(source=publish) 公开访问快照
```

这样用户发布后继续编辑并保存草稿，也不会自动影响线上公开页面，直到再次点击发布。

## 5. 演示时建议话术

可以按这个顺序讲：

> 我先登录进入项目面板，然后创建一个项目和页面。进入页面后，编辑器会先从后端读取页面 schema，再写入 Zustand 恢复组件树。左侧是物料区，中间是画布，右侧是属性设置面板。拖拽组件本质上是往组件树里添加节点，修改属性就是更新节点的 props 或 styles。
>
> 点击保存时，前端会把当前组件树包装成 PageSchema，通过接口保存到 PostgreSQL。后端除了更新当前 Page.schema，还会生成 PageVersion 历史版本。所以我可以打开版本历史，看到每次保存的快照。如果选择某个旧版本回滚，后端会把旧的 PageVersion.schema 恢复到 Page.schema，同时再生成一条 rollback 版本。删除历史版本时，只删除 PageVersion，不影响当前 Page.schema。
>
> 发布时，后端会基于当前 Page.schema 创建一条 publish 来源的 PageVersion 快照，并把页面的 publishedVersionId 指向这条快照。访客访问 /publish/:publicId 时不需要登录，看到的是发布快照，不会被后续草稿编辑影响。
>
> 这个项目的重点是从一个前端低代码 demo，逐步完善成有登录、项目、页面、保存、恢复、版本、回滚和发布访问能力的产品闭环。
