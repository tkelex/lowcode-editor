# 企业级工程规范化计划

## 目标

把当前项目从“功能逐步堆叠的学习型同仓项目”，整理成“边界清晰、可扩展、可测试、可部署”的企业级工程结构。

本计划只规划代码组织和工程治理，不立即大规模移动文件。后续实施时应按阶段逐步迁移，每个阶段都保证：

- 前端可启动、可构建。
- 后端可启动、可构建。
- 现有保存、预览、发布、版本回滚链路不回归。
- 文档、脚本、路径引用同步更新。

## 当前主要问题

### 前端

- `src/App.tsx` 同时承担登录态、项目面板、编辑器、公开发布页分发，应用入口职责偏重。
- `src/api/` 类型、请求方法和业务页面耦合较松，未来接口增多后容易散。
- `src/editor/` 已形成独立领域，但内部还有历史拼写和重复文件：
  - `components/Preivew` 与 `components/Preview`
  - `components/Sourse` 与 `components/Source`
  - `useMaterailDrop.ts` 与 `useMaterialDrop.ts`
- 事件系统、schema 校验、物料注册已经变复杂，需要进一步拆清楚边界。
- 当前缺少统一测试目录和测试策略。

### 后端

- NestJS 模块结构已经基本正确，但后续如果增加资产、成员、发布记录、模板等模块，需要统一领域边界。
- 页面 schema 校验目前主要在前端，后端仍偏 JSON 透传，未来需要服务端 schema contract。
- DTO、响应类型、错误码和前端类型还没有统一生成或共享。
- 缺少测试分层：单元测试、集成测试、API smoke test。

### 全仓

- 当前是前后端同仓，但还不是标准 monorepo。
- 缺少统一的 `scripts/`、`infra/`、`packages/` 分层。
- 缺少 CI 配置、提交校验、自动格式化策略。
- 文档已经归类，但代码结构还没有同步产品化。

## 总体原则

1. **先建边界，再搬文件**：先定义目标结构和模块职责，再按模块迁移。
2. **先低风险整理，再高风险重构**：先清理重复文件、抽公共类型，再调整入口和目录。
3. **保留兼容层**：历史 typo API 不直接删除，先用 re-export 或兼容文件过渡。
4. **按领域组织代码**：业务功能按 auth、projects、pages、editor、publish 等领域组织。
5. **公共能力下沉**：请求、类型、工具、常量、配置、错误处理放到 shared 层。
6. **每个阶段可验证**：阶段完成后必须运行构建，重要链路需要手测或自动化验证。

## 目标仓库结构

长期目标可以演进为轻量 monorepo：

```text
lowcode-editor/
  apps/
    web/                     # React 前端应用
    server/                  # NestJS 后端应用
  packages/
    shared/                  # 前后端共享类型、常量、schema
    lowcode-schema/          # 页面 schema、事件动作 schema、迁移器
    ui/                      # 后续可选：业务 UI 组件库
  docs/                      # 已分类文档
  scripts/                   # 自动化脚本、smoke test、数据初始化
  infra/                     # Docker、Nginx、部署、CI 配置
  package.json
```

短期不建议马上移动到 `apps/`，因为会影响 Vite、Nest、路径、脚本和部署。第一阶段可以先在现有结构内整理边界。

## 前端目标结构

短期目标结构：

```text
src/
  app/
    App.tsx                  # 应用路由/视图编排
    providers/               # DnD、主题、全局错误边界等 Provider
    routes/                  # 路由识别和页面分发
  shared/
    api/                     # axios 实例、请求工具、错误处理
    config/                  # 环境变量、运行时配置
    constants/               # 全局常量
    hooks/                   # 通用 hooks
    types/                   # 通用类型
    utils/                   # 通用工具函数
  features/
    auth/
    projects/
    publish/
  editor/
    components/
    materials/
    events/
    schema/
    stores/
    hooks/
    runtime/
    setting/
```

### 前端分层规则

| 层 | 作用 | 允许依赖 |
| --- | --- | --- |
| `app` | 应用装配、路由、全局 Provider | `features`、`editor`、`shared` |
| `features` | 业务页面和业务流程 | `shared`，必要时依赖 `editor` |
| `editor` | 低代码编辑器领域 | `shared` |
| `shared` | 无业务领域的公共能力 | 不依赖 `features` 和 `editor` |

### 编辑器内部建议

```text
src/editor/
  components/                # 编辑器布局组件：Header/EditArea/Material/Outline/Source
  materials/                 # 物料 dev/prod 实现
  registry/                  # 物料注册表，后续从 stores/component-config.tsx 迁出
  events/                    # 事件动作类型、运行器、动作实现
  schema/                    # schema 校验、迁移、默认 schema
  stores/                    # Zustand store
  runtime/                   # 预览/发布运行态渲染
  setting/                   # 属性、样式、事件配置面板
  hooks/                     # 编辑器专用 hooks
  types/                     # 编辑器类型
```

后续可以把当前 `src/editor/events.ts` 拆成：

```text
events/
  types.ts
  normalize.ts
  createEventData.ts
  runActions.ts
  actions/
    toast.ts
    url.ts
    confirm.ts
    condition.ts
    http.ts
    componentAction.ts
    updateComponent.ts
    custom.ts
```

## 后端目标结构

当前 NestJS 结构可以保留，但新增模块时统一使用领域模块：

```text
server/src/
  main.ts
  app.module.ts
  common/
    decorators/
    filters/
    guards/
    interceptors/
    pipes/
    errors/
  config/
    env.ts
    validation.ts
  prisma/
  modules/
    auth/
    users/
    projects/
    pages/
    page-versions/           # 后续可从 pages 中拆出
    publish/                 # 后续发布记录、取消发布、发布流水
    assets/                  # 后续素材上传
    members/                 # 后续项目成员权限
    templates/               # 后续页面模板
```

### 后端模块规范

每个业务模块建议保持：

```text
modules/<domain>/
  <domain>.module.ts
  <domain>.controller.ts
  <domain>.service.ts
  dto/
  entities/                  # 如需要响应模型或领域对象
  constants.ts
  errors.ts
```

### 后端分层规则

| 层 | 职责 | 禁止事项 |
| --- | --- | --- |
| Controller | 接收请求、解析参数、调用 Service | 写复杂业务逻辑 |
| Service | 业务规则、权限校验、事务编排 | 直接暴露 Prisma 错误给前端 |
| DTO | 请求参数校验 | 放业务逻辑 |
| Prisma | 数据库访问 | 在 Controller 中直接使用 |
| Common | 通用 Guard、Filter、Pipe、Decorator | 混入具体业务逻辑 |

## 共享类型和 Schema

中期建议新增：

```text
packages/shared/
  src/
    api/
    types/
    constants/

packages/lowcode-schema/
  src/
    component.ts
    page-schema.ts
    event-action.ts
    validate.ts
    migrate.ts
```

目标：

- 前后端共享 `PageSchema`、`ComponentSchema`、`EventAction` 类型。
- 前端保存前和后端入库前使用同一套 schema 校验。
- 后续旧 schema migration 集中维护，不散落在组件或运行态里。

当前已落地 `packages/lowcode-schema`，作为前后端共享的 schema 边界。它集中维护页面 schema 类型、默认 schema、内置物料父子关系、组件树校验和 schema migration；前端通过兼容出口继续服务编辑器，后端在入库、发布和回滚前统一调用。

## 命名规范

### 目录命名

- React 组件目录：PascalCase，例如 `ProjectDashboard`。
- 普通工具目录：kebab-case 或小写复数，例如 `utils`、`constants`。
- NestJS 模块目录：小写复数，例如 `projects`、`pages`、`assets`。
- 文档目录：保留中文编号目录。

### 文件命名

- React 组件：`ComponentName.tsx` 或目录内 `index.tsx`，同一模块内保持一致。
- hooks：`useXxx.ts`。
- store：`xxx.store.ts` 或保留当前 `stores/xxx.tsx`，迁移时统一。
- API：`xxx.api.ts` 或按当前 `api/xxx.ts` 过渡。
- DTO：`create-xxx.dto.ts`、`update-xxx.dto.ts`。
- 测试：`*.test.ts`、`*.spec.ts`。

### 历史 typo 处理

这些历史 API 不能直接删除：

- `useComponetsStore`
- `useMaterailDrop`
- `components/Preivew`
- `components/Sourse`

建议分三步处理：

1. 新增正确命名文件作为主实现。
2. 旧文件只做 re-export，并标记 deprecated。
3. 全项目迁移引用后，再在独立版本中删除旧文件。

## 配置和环境变量

目标结构：

```text
.env.example
server/.env.example
src/shared/config/
server/src/config/
```

规范：

- 所有环境变量必须有 example。
- 前端只允许暴露 `VITE_` 前缀变量。
- 后端启动时校验必要环境变量。
- 不在业务代码中直接散落读取环境变量。

## 测试规划

### 前端测试

建议逐步引入：

- Vitest：测试纯函数、schema 校验、事件动作工具。
- React Testing Library：测试设置面板、事件配置 UI。
- Playwright：测试编辑器关键链路。

优先测试：

1. `validateComponentTree`
2. 事件动作 URL normalize
3. 旧事件 schema normalize
4. 源码面板非法 JSON 不写入组件树
5. 保存/发布前校验阻断非法 schema

### 后端测试

建议逐步引入：

- Jest 单元测试：Service 业务逻辑。
- Supertest 集成测试：Auth、Projects、Pages API。
- API smoke script：注册、登录、创建项目、创建页面、保存、发布、公开读取。

优先测试：

1. Auth 注册登录
2. Project owner 权限
3. Page owner 权限
4. 版本回滚不能跨页面
5. 删除版本不影响当前 Page.schema
6. 发布快照和草稿隔离

## 不建议现在做的事

- 不建议马上把整个仓库移动到 `apps/web` 和 `apps/server`，风险较高。
- 不建议一次性重命名所有 typo 文件，容易制造大量冲突。
- 不建议过早抽很多 packages，当前共享边界还需要稳定。
- 不建议为了“企业级”引入复杂微服务、权限中台或多租户框架。
- 不建议在没有测试保护时大规模重构编辑器渲染链路。

## 第一阶段建议执行清单

优先做这些低风险工作：

1. 建立 `src/app`，逐步拆小 `App.tsx`。
2. 确认并删除未使用的重复文件或改为 re-export。
3. 把事件动作运行器从单文件拆成 `editor/events/` 目录。
4. 把物料注册表从 `stores` 概念中迁到 `editor/registry`。
5. 新增 `scripts/smoke`，固化核心 API 验证。
6. 新增 `npm run check`，统一本地检查命令。
7. 更新 `docs/05-开发/编码规范.md`，让规范和代码结构保持一致。

## 验收标准

当工程规范化第一阶段完成后，应满足：

- 新人能通过 `docs/README.md` 和本计划理解目录边界。
- 新功能知道应该放在 `app`、`features`、`editor`、`shared`、`server/modules` 的哪个位置。
- 旧 typo API 有明确兼容策略。
- 前后端核心业务闭环没有回归。
- 项目具备统一检查命令。
- 后续新增资产、模板、成员权限、数据源等模块时，不需要重新讨论基本目录结构。
