# 编码规范

## 通用原则

- 优先实现明确需求，不做超出当前阶段的抽象。
- 前后端都使用 TypeScript。
- 重要业务数据结构需要显式类型。
- 不提交真实密钥、token、数据库连接串。
- 错误要返回清晰信息，但不要泄露内部堆栈和敏感配置。

## 前端规范

### 目录

- `src/app/`：应用装配层，负责 Provider、路由识别、顶层视图分发。
- `src/shared/api/`：接口请求主实现，包括 axios 实例、业务 API 和前端 API 类型。
- `src/api/`：历史兼容层，只做 re-export，新代码不要继续从这里引入。
- `docs/02-架构/模块边界与拆分规范.md`：模块拆分和依赖方向的长期规则，新增大功能前先对照这里放置代码。
- `src/features/`：业务模块，如 auth、projects、publish。
- `src/features/<feature>/model/`：功能内类型、展示字典和纯业务规则；页面组件不要长期堆积这些常量。
- `src/editor/`：保留现有编辑器核心。
- `src/editor/stores/`：编辑器状态。
- `src/editor/components/Preview`、`src/editor/components/Source`、`src/editor/hooks/useMaterialDrop.ts`：后续新增代码优先使用正确命名入口。

### 状态

- 编辑器运行时状态继续使用 Zustand。
- 后端返回的页面 schema 加载后写入编辑器 store。
- 不直接在组件中散落 localStorage 读写。
- Token 可以先集中封装在 auth API/state 中，后续再升级更完整的会话管理。

### API

- axios 实例统一放在 `src/shared/api/http.ts`。
- 业务接口按模块拆分，例如 `auth.ts`、`projects.ts`、`pages.ts`。
- 新代码统一从 `src/shared/api/*` 引入 API 和类型；`src/api/*` 只保留旧路径兼容。
- 新增 API 时同步 `src/shared/api/index.ts`；如已有旧路径使用场景，再补 `src/api/<name>.ts` 兼容导出。
- 组件中不要直接拼接复杂 URL。
- 请求失败必须给用户明确提示。

### 编辑器

- 保留现有公开 typo API：`useComponetsStore`、`useMaterailDrop`、`components/Preivew`、`components/Sourse`。
- 历史 typo 文件只做兼容导出，新代码使用 `useMaterialDrop`、`components/Preview`、`components/Source`。
- 新增 store action 时保持现有嵌套对象更新模式：修改后替换顶层 `components` 数组触发更新。
- dev 物料根节点需要暴露 `data-component-id`。

## 后端规范

### 目录

- `server/src/modules/<domain>/`：业务模块。
- `server/src/common/`：guard、decorator、filter、pipe 等通用能力。
- `server/src/prisma/`：PrismaService 和数据库连接。

### API

- Controller 只做参数接收和返回，不写复杂业务逻辑。
- Service 负责业务规则和数据库访问。
- DTO 使用 `class-validator` 做输入校验。
- 受保护接口必须使用 JWT Guard。
- 涉及项目/页面的数据读取必须校验 owner。

### 数据库

- 使用 Prisma schema 管理模型。
- 生产数据不使用 `db push` 替代迁移；开发阶段可临时使用。
- 页面 schema 使用 JSON 字段保存。
- 重要记录保留 `createdAt`、`updatedAt`。

### 错误处理

- 参数错误返回 400。
- 未登录返回 401。
- 无权限或访问他人资源返回 403 或 404。
- 资源不存在返回 404。
- 不把数据库错误原样暴露给前端。

## 命名

- 前端组件使用 PascalCase。
- hooks 使用 `useXxx`。
- API 函数使用动词开头，例如 `createProject`、`updatePage`。
- DTO 使用 `CreateXxxDto`、`UpdateXxxDto`。
- Prisma model 使用单数 PascalCase。

## 提交前检查

- 常规改动：优先运行 `npm run check`，统一检查前端 lint、前端构建和后端构建。
- 架构或目录边界改动：至少运行 `npm run check:architecture`。
- 前端专项改动：至少运行 `npm run lint` 和 `npm run build`。
- 后端专项改动：至少运行 `npm run build --prefix server`，有 lint 需求时运行 `npm run lint --prefix server`。
- 涉及数据库：确认 Prisma schema 和环境变量说明同步更新。
