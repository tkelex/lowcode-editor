# 部署与运维指南

## 目标

本文档合并原来的上线攻略、生产部署流程、发布回滚手册和运行排障说明，作为部署和运维的统一入口。更细的数据库备份恢复只保留在 `docs/05-开发/数据库备份与恢复.md`。

如果前端准备托管到 Netlify，请先看 `docs/05-开发/GitHub与Netlify上线指南.md`。Netlify 只负责前端静态资源，NestJS 后端和 PostgreSQL 仍需要独立部署。

## 部署前准备

服务器需要：

- Docker 和 Docker Compose。
- 一个真实域名，例如 `lowcode.example.com`。
- 可用的 80/443 入口，或由外层网关代理到本项目 web 容器。
- 一套数据库备份策略。
- 一个准备授予平台管理员的邮箱。

上线前必须确认：

- `JWT_SECRET` 是强随机密钥，不使用示例值。
- 数据库密码足够强。
- `FRONTEND_ORIGIN` 是真实前端域名。
- `VITE_API_BASE_URL` 使用 `/api` 或可信 API 域名。
- `VITE_LOWCODE_HTTP_ALLOWED_ORIGINS` 已收口低代码 HTTP 动作可访问域名。
- `UPLOAD_DIR` 指向持久化目录。
- `.env`、真实密钥、数据库密码不会提交到 Git。

## 生产环境变量

复制生产环境模板：

```powershell
Copy-Item infra/docker/.env.prod.example infra/docker/.env
Copy-Item server/.env.production.example server/.env.production
```

至少修改：

- `infra/docker/.env`
- `server/.env.production`

重点字段：

- `POSTGRES_PASSWORD`
- `DATABASE_URL`
- `JWT_SECRET`
- `FRONTEND_ORIGIN`
- `UPLOAD_DIR`
- `WEB_PORT`
- `VITE_LOWCODE_HTTP_ALLOWED_ORIGINS`

如果使用 `infra/docker/docker-compose.prod.example.yml`，主要运行变量会从 `infra/docker/.env` 注入。`server/.env.production` 主要用于非容器启动或排查。

## 首次部署

构建并启动：

```powershell
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml up -d --build
```

查看容器状态：

```powershell
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml ps
```

健康检查：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/ops/check-production.ps1 -BaseUrl "https://你的域名"
```

如果还没有 HTTPS，可以先用：

```text
http://服务器IP:WEB_PORT/api/health
```

`server` 容器启动时会先执行 `npm run prisma:deploy`，再启动 NestJS 服务。

## 管理员账号

先注册一个普通用户，再在生产容器中授予管理员：

```powershell
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml exec server npm run admin:grant -- --email admin@example.com
```

重新登录后，项目面板会出现“管理后台”入口。生产环境不要开放“注册管理员”接口，管理员账号应通过脚本或内部运维流程授予。

## 发布升级流程

每次升级建议按这个顺序：

1. 拉取新代码并记录 commit。
2. 备份数据库。
3. 检查 `server/prisma/migrations` 是否新增 migration。
4. 构建并启动新镜像。
5. 等待 `/api/health` 正常。
6. 运行生产健康检查。
7. 登录后台做上线验收。

常用命令：

```powershell
git pull
git rev-parse HEAD
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml up -d --build
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml logs -f server
powershell -ExecutionPolicy Bypass -File scripts/ops/check-production.ps1 -BaseUrl "https://你的域名"
```

上线前数据库备份示例：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/db/backup-postgres.ps1 -DockerComposeService postgres -ContainerDatabaseUrl "postgresql://lowcode_app:你的密码@localhost:5432/lowcode_editor?schema=public"
```

完整备份和恢复说明见 `docs/05-开发/数据库备份与恢复.md`。

## 上线验收

最少确认：

- 首页能打开。
- `GET /api/health` 正常。
- 能注册或登录。
- 能创建项目和页面。
- 能进入编辑器。
- 能保存页面并重新打开恢复。
- 能发布页面并访问公开页。
- 管理后台能打开。
- 管理员能取消发布、禁用用户或禁用项目。
- 审计日志能看到关键操作。

完整清单见 `docs/06-测试与验收/上线验收清单.md`。

## 日志和排障

查看日志：

```powershell
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml logs -f server
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml logs -f web
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml logs -f postgres
```

后端请求日志是 JSON 行，包含：

- `requestId`
- `method`
- `path`
- `status`
- `durationMs`
- `timestamp`

错误响应也会带 `requestId`。用户反馈问题时，优先用排障编号搜索 server 日志：

```powershell
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml logs server | Select-String "requestId"
```

日志不会记录请求体、Authorization 或 query 中的敏感信息。生产环境不会输出完整堆栈。

## 回滚流程

如果只是应用代码问题，且没有不兼容数据库 migration：

```powershell
git checkout <上一版commit>
docker compose --env-file infra/docker/.env -f infra/docker/docker-compose.prod.example.yml up -d --build
powershell -ExecutionPolicy Bypass -File scripts/ops/check-production.ps1 -BaseUrl "https://你的域名"
```

如果已经执行不兼容 migration：

1. 不要直接覆盖生产库。
2. 先保留当前数据库和上传目录。
3. 优先恢复发布前备份。
4. 恢复后重新检查项目、页面、版本历史、公开发布页和审计日志。

数据库 migration 通常不是自动可逆的。正式生产回滚前必须先有备份。

## 常见问题

### 后端启动时报 JWT_SECRET must be strong

生产环境仍使用弱密钥或示例密钥。重新生成强密钥并更新生产环境变量。

### 后端启动时报 FRONTEND_ORIGIN must not use localhost

生产环境不能继续使用 `http://localhost:5173`。改成真实域名，例如：

```text
FRONTEND_ORIGIN=https://lowcode.example.com
```

### 前端能打开但 API 404

检查：

- Nginx `/api/` 是否代理到 `server:3000/api/`。
- `VITE_API_BASE_URL` 是否为 `/api`。
- `server` 容器健康检查是否通过。

### CORS 错误

检查：

- 后端 `FRONTEND_ORIGIN` 是否等于真实前端域名。
- 浏览器访问的前端域名是否和 `FRONTEND_ORIGIN` 完全一致。
- 生产环境不要继续使用 `localhost`。

### 数据库连接失败

检查：

- `POSTGRES_USER`、`POSTGRES_PASSWORD`、`POSTGRES_DB` 是否一致。
- `DATABASE_URL` 是否指向 `postgres:5432`。
- `postgres` 容器是否 healthy。
- migration 是否执行成功。

### 上传文件丢失

检查：

- `uploads_data` volume 是否存在。
- `UPLOAD_DIR` 是否为持久化目录。
- 是否误执行了 `docker compose down -v`。

### 公开页白屏

检查：

- 页面是否仍处于已发布状态。
- 项目是否被管理员禁用。
- 发布快照 schema 是否能通过迁移和校验。
- 公开页必须保持 `allowCustomJS=false`。

## 运维习惯

- 所有发布先记录版本号和数据库备份文件。
- 所有异常先记录 `requestId`。
- 所有升级先确认 migration 是否兼容。
- 所有回滚先确认数据库是否需要恢复。
- 不要在没有备份时升级生产库。
