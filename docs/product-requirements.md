# 低代码编辑器需求文档

## 1. 产品定位

本项目是一个面向前端页面搭建场景的低代码编辑器，目标是让用户通过可视化拖拽、属性配置和事件配置，快速搭建可预览、可保存、可发布的业务页面。

它不是一个通用的“万能低代码平台”，第一阶段应聚焦在中后台业务页面、表单页面、数据展示页面和简单交互页面。

## 2. 服务对象

### 2.1 主要用户

#### 前端开发者

前端开发者是第一阶段最重要的用户。

他们使用低代码编辑器的目的不是完全替代编码，而是减少重复页面开发工作，例如表单、按钮、表格、弹窗、布局容器等常见页面结构。

他们关心：

- 页面结构是否清晰
- 组件配置是否灵活
- 生成或保存的数据结构是否可维护
- 是否方便调试和扩展组件
- 是否能接入真实接口和业务系统

#### 后台系统搭建者

这类用户可能是开发者、技术支持、实施人员或懂一点技术的业务人员。

他们希望不写代码或少写代码，快速搭建管理页面、配置页面、活动页面或简单业务页面。

他们关心：

- 操作是否简单
- 配置是否直观
- 页面是否能保存和复用
- 是否能预览最终效果
- 出错后是否容易恢复

### 2.2 次要用户

#### 产品经理 / 设计人员

他们可能使用编辑器制作页面原型，验证页面结构和业务流程。

他们关心：

- 是否能快速表达页面想法
- 是否能预览交互
- 是否方便给开发或团队成员查看

#### 管理员

管理员主要负责用户、项目、权限和发布内容管理。

他们关心：

- 谁创建了项目
- 谁修改了页面
- 哪个版本被发布
- 是否可以回滚错误发布

## 3. 最重要的事情

低代码编辑器最重要的事情不是“组件数量多”，而是：

> 用户能够稳定地创建页面、保存页面、再次打开页面，并可靠地发布一个可访问的版本。

因此，第一阶段最重要的产品闭环是：

```text
创建项目 → 创建页面 → 拖拽组件 → 配置属性/样式/事件 → 保存 → 预览 → 发布 → 再次编辑 → 版本回滚
```

如果这个闭环不稳定，即使组件很多，也不能算是可上线产品。

## 4. 核心问题

本产品要解决的问题：

1. 重复业务页面开发效率低。
2. 页面配置缺少统一存储和版本管理。
3. 普通用户无法独立搭建简单页面。
4. 页面发布依赖开发手动处理，流程不透明。
5. 配置错误后缺少回滚能力。

## 5. 核心价值

### 5.1 对开发者

- 减少重复 CRUD 页面开发。
- 用组件 schema 描述页面，便于保存、复制、版本管理和渲染。
- 通过物料系统扩展业务组件。
- 降低后续搭建相似页面的成本。

### 5.2 对业务或实施人员

- 通过拖拽和配置完成页面搭建。
- 不依赖开发即可调整文案、布局和部分交互。
- 可以预览页面真实效果。
- 出错后可以回滚到历史版本。

### 5.3 对项目维护者

- 页面数据统一存储。
- 发布记录可追踪。
- 页面变更有历史版本。
- 后续可以扩展权限、模板、素材库和组件市场。

## 6. MVP 范围

MVP 的目标是做出一个“可保存、可预览、可发布”的低代码页面编辑器，而不是完整商业平台。

### 6.1 必须包含

#### 用户系统

- 用户注册
- 用户登录
- 获取当前用户信息
- 退出登录

#### 项目管理

- 创建项目
- 查看项目列表
- 编辑项目名称和描述
- 删除项目

#### 页面管理

- 创建页面
- 查看页面列表
- 编辑页面名称和路径
- 删除页面
- 打开页面进入编辑器

#### 编辑器能力

- 拖拽添加组件
- 移动已有组件
- 删除组件
- 选中组件
- 配置组件属性
- 配置组件样式
- 配置组件事件
- 查看组件大纲
- 查看页面 schema 源码
- 编辑模式和预览模式切换

#### 保存能力

- 保存当前页面 schema
- 刷新页面后能从后端恢复
- 保存时显示成功或失败状态
- 防止用户误以为已经保存

#### 版本能力

- 手动保存页面版本
- 查看页面历史版本
- 回滚到指定版本

#### 发布能力

- 发布当前页面版本
- 查看发布记录
- 访问发布后的页面

### 6.2 可以延后

- 多人实时协作
- 复杂权限体系
- 组件市场
- 自定义代码组件上传
- 复杂工作流审批
- 多环境发布审批
- A/B 测试
- 页面性能分析平台化
- 国际化
- 复杂模板市场

## 7. 关键用户场景

### 场景一：开发者创建一个后台表单页面

1. 用户登录系统。
2. 创建一个项目。
3. 在项目中创建页面。
4. 拖入表单、表单项、按钮、弹窗等组件。
5. 配置表单字段、按钮文案和提交事件。
6. 点击预览检查效果。
7. 保存页面。
8. 发布页面。
9. 通过发布地址访问页面。

成功标准：用户不直接修改代码，也能完成一个基础表单页面。

### 场景二：用户修改已发布页面

1. 用户打开已有项目。
2. 进入某个已发布页面。
3. 修改按钮文案或页面布局。
4. 保存为新版本。
5. 预览确认。
6. 发布新版本。
7. 发布记录中能看到新旧版本。

成功标准：用户能安全修改页面，并知道当前发布的是哪个版本。

### 场景三：发布后发现配置错误并回滚

1. 用户发布新版本后发现页面配置错误。
2. 打开历史版本列表。
3. 选择上一个稳定版本。
4. 执行回滚。
5. 重新发布。

成功标准：错误配置不会造成不可恢复的问题。

### 场景四：用户查看页面结构

1. 用户在编辑器中搭建页面。
2. 在左侧切换到大纲视图。
3. 查看组件层级。
4. 点击大纲中的组件并定位到画布。

成功标准：页面复杂后，用户仍然能理解和管理组件结构。

## 8. 功能需求

### 8.1 项目与页面

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 创建项目 | P0 | 用户必须先有项目容器 |
| 项目列表 | P0 | 进入系统后的主要入口 |
| 创建页面 | P0 | 每个项目下可以有多个页面 |
| 页面列表 | P0 | 查看和进入页面 |
| 页面路径配置 | P1 | 发布后用于访问页面 |
| 删除项目/页面 | P1 | 需要二次确认 |

### 8.2 编辑器

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 物料面板 | P0 | 展示可拖拽组件 |
| 画布渲染 | P0 | 显示当前页面结构 |
| 拖拽添加组件 | P0 | 核心搭建能力 |
| 组件选中 | P0 | 配置属性的前提 |
| 属性配置 | P0 | 修改组件基础配置 |
| 样式配置 | P0 | 修改宽高等样式 |
| 事件配置 | P0 | 配置跳转、消息、组件方法等动作 |
| 大纲树 | P1 | 管理复杂页面结构 |
| 源码视图 | P1 | 查看 schema，方便开发者调试 |
| 组件删除 | P0 | 基础编辑能力 |
| 组件移动 | P1 | 优化搭建体验 |

### 8.3 保存与版本

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 保存页面 | P0 | 把 schema 持久化到后端 |
| 加载页面 | P0 | 从后端恢复编辑状态 |
| 保存状态提示 | P0 | 明确保存成功/失败 |
| 历史版本 | P1 | 保存重要节点 |
| 回滚版本 | P1 | 防止错误不可恢复 |

### 8.4 发布

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 发布页面 | P0 | 生成可访问页面 |
| 发布记录 | P1 | 追踪谁在何时发布了哪个版本 |
| 访问发布页面 | P0 | 上线产品必须具备 |
| 重新发布历史版本 | P1 | 配合回滚使用 |

### 8.5 权限

MVP 阶段可以先做简单权限：

- 项目创建者是 owner。
- 只有 owner 可以编辑和删除项目。
- 只有登录用户可以创建项目。

复杂的成员管理、角色权限可以放到后续版本。

## 9. 非功能需求

### 9.1 稳定性

- 页面 schema 保存失败时必须提示用户。
- 发布失败时不能影响已有已发布版本。
- 回滚操作需要二次确认。

### 9.2 安全性

- 所有项目和页面接口需要登录态。
- 用户只能访问自己有权限的项目。
- 自定义 JS 能力需要谨慎处理，至少在 MVP 中限制其使用范围或仅面向可信用户。
- 上传文件需要限制类型和大小。

### 9.3 性能

- 编辑器在 100 个组件规模下应保持可编辑。
- 页面 schema 保存不应阻塞用户操作太久。
- 发布页首屏渲染应尽量轻量。

### 9.4 可维护性

- 页面 schema 结构需要保留版本号。
- 后端接口需要有清晰错误码。
- 组件物料需要有统一注册规范。
- 发布记录和版本记录不能混用。

## 10. 页面 schema 要求

页面 schema 是低代码平台的核心资产。

建议 schema 至少包含：

- schemaVersion
- pageId
- components
- metadata

示例：

```json
{
  "schemaVersion": "1.0.0",
  "pageId": 1,
  "components": [
    {
      "id": 1,
      "name": "Page",
      "props": {},
      "styles": {},
      "desc": "页面",
      "children": []
    }
  ],
  "metadata": {
    "updatedAt": "2026-04-28T00:00:00.000Z"
  }
}
```

当前前端已有 `components` 树，后续可以先把它作为 schema 的核心字段保存。

## 11. 成功指标

第一阶段不建议用复杂商业指标，先用产品闭环指标：

- 用户能在 10 分钟内创建并发布一个简单页面。
- 页面刷新后内容不丢失。
- 已发布页面可以通过 URL 访问。
- 页面错误配置可以通过版本回滚恢复。
- 一个项目下可以管理多个页面。

## 12. 第一版验收标准

第一版上线前至少满足：

1. 用户可以注册和登录。
2. 用户可以创建项目和页面。
3. 页面编辑器可以从后端加载 schema。
4. 页面编辑器可以保存 schema 到后端。
5. 用户可以预览页面。
6. 用户可以发布页面。
7. 发布后的页面可以访问。
8. 用户可以查看历史版本。
9. 用户可以回滚到历史版本。
10. 基础接口有权限校验。

## 13. 产品边界

第一阶段应坚持以下边界：

- 先服务中后台页面，不做所有类型的网站。
- 先服务单人编辑，不做多人实时协作。
- 先做稳定保存和发布，不追求复杂动画和高级设计能力。
- 先做固定物料体系，不做在线安装第三方组件。
- 先做简单权限，不做企业组织架构。

## 14. 后续演进方向

当 MVP 稳定后，可以考虑：

1. 模板市场：用户可以从模板快速创建页面。
2. 业务组件库：沉淀项目常用组件。
3. 数据源配置：通过配置连接后端接口。
4. 权限协作：项目成员、角色、只读访问。
5. 发布环境：区分测试、预发、生产。
6. 操作日志：记录页面关键变更。
7. 页面复制：快速复用已有页面。
8. AI 辅助搭建：通过自然语言生成页面初稿。

## 15. 当前项目对应关系

当前前端已经具备部分 MVP 基础：

- 已有编辑/预览模式。
- 已有组件树状态。
- 已有物料注册机制。
- 已有属性、样式、事件配置。
- 已有大纲和源码入口。
- 已有预览事件执行逻辑。

接下来最关键的产品化工作是：

1. 增加后端持久化。
2. 增加用户、项目、页面概念。
3. 增加保存、版本和发布流程。
4. 明确页面 schema 格式。
5. 建立上线部署流程。
