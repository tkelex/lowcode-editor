# 模块边界与拆分规范

本文用于约束本项目的前端、后端、数据库、共享协议和功能模块边界。目标不是一次性搬目录，而是先让边界清楚、依赖可检查，再逐步把大文件和混合职责拆开。

## 参考原则

- 前端按 Feature-Sliced Design 的分层思想落地：越靠上的层负责装配，越靠下的层越通用；业务功能放在 slice 内隔离。
- 后端沿用 NestJS module 组织业务，每个领域模块通过 controller、service、dto、module 形成边界。
- 仓库保持轻量 monorepo：前端、后端、共享 schema、数据库迁移和基础设施各自独立，只有共享协议层允许被前后端共同依赖。

参考资料：
- Feature-Sliced Design layers: https://feature-sliced.github.io/documentation/docs/reference/layers
- NestJS workspace / monorepo: https://docs.nestjs.com/cli/monorepo

## 顶层边界

```text
lowcode-editor/
  src/                         前端应用，只依赖 packages/lowcode-schema 和浏览器侧依赖
  server/                      NestJS 后端，只通过 Prisma 访问数据库
  server/prisma/               数据库模型和 migrations，不被前端直接引用
  packages/lowcode-schema/     前后端共享 schema、事件 action、迁移和校验
  scripts/                     测试、运维、架构检查脚本
  infra/                       Docker、Nginx、部署模板
  docs/                        项目文档
```

依赖方向：

```text
src  ───────────────┐
                    ├──> packages/lowcode-schema
server ─────────────┘

server ───> server/prisma ───> PostgreSQL
src ───> HTTP API ───> server
```

禁止方向：

- `src` 不直接 import `server` 或 `server/prisma`。
- `server` 不直接 import `src`。
- `packages/lowcode-schema` 不依赖 `src` 或 `server`。
- `src/api` 只做旧路径兼容 re-export，新代码统一从 `src/shared/api` 引入。

## 前端模块边界

当前前端不强制完整迁移到 FSD 的所有层，先采用项目已有结构：

```text
src/app/          应用装配：Provider、路由识别、顶层视图分发、全局错误兜底
src/features/     业务功能入口：auth、projects、admin、publish
src/editor/       低代码编辑器领域：画布、物料、配置面板、运行态预览
src/shared/       跨业务共享能力：api、通用组件、工具
src/api/          旧 API 路径兼容层，只允许 re-export
```

前端依赖规则：

- `app` 可以组合 `features`、`editor`、`shared`。
- `features` 可以使用 `shared`，必要时可以调用 `editor` 的公开运行态能力，例如公开页渲染 `Preview`。
- `editor` 可以使用 `shared/api` 的类型和少量 API，但编辑器核心状态、物料、事件执行不要反向依赖业务页面。
- `shared` 不能依赖 `app`、`features` 或 `editor`。
- 新 API 封装只放在 `src/shared/api`；`src/api` 不写业务逻辑。

## 后端模块边界

后端继续按 NestJS 领域模块拆分：

```text
server/src/common/       guard、decorator、filter、middleware、error、logging
server/src/config/       环境变量校验
server/src/prisma/       PrismaService
server/src/modules/auth/
server/src/modules/users/
server/src/modules/projects/
server/src/modules/pages/
server/src/modules/assets/
server/src/modules/templates/
server/src/modules/admin/
server/src/modules/audit/
server/src/modules/health/
```

后端依赖规则：

- Controller 只负责路由、参数、鉴权装饰器和响应编排。
- Service 负责业务规则、权限校验和数据库访问。
- DTO 只负责输入形状和校验。
- 跨模块复用优先通过明确 service 注入，不从其它模块深层文件绕路读取数据。
- 项目、页面、素材、模板等业务数据访问必须走权限检查；不能只依赖前端隐藏按钮。

## 数据库边界

- 数据库结构只通过 `server/prisma/schema.prisma` 和 migrations 管理。
- 前端永远不读取 Prisma schema、migration 或数据库连接配置。
- 改 Prisma model 后必须生成 migration，并同步后端 DTO/service、前端 `src/shared/api/types.ts` 和接口文档。
- 页面 schema 的 JSON 内容由 `packages/lowcode-schema` 负责迁移和校验，数据库只负责持久化。

## 共享协议边界

`packages/lowcode-schema` 是唯一允许前后端共同依赖的业务协议层，适合放：

- 页面 schema 和组件 schema 类型。
- 事件 action 类型。
- 默认 schema。
- schema 迁移。
- schema 校验。
- 物料父子关系规则。
- URL、HTTP action、安全表达式等运行时可共享纯函数。

不适合放：

- React 组件。
- NestJS service/controller。
- axios、Prisma、localStorage、DOM 操作。
- 任何需要真实环境变量或数据库连接的逻辑。

## 功能隔离规则

新增功能时先判断属于哪一类：

| 功能类型 | 放置位置 |
| --- | --- |
| 登录、项目列表、管理员后台、公开发布页 | `src/features/<feature>` |
| 编辑器画布、物料、设置面板、预览运行态 | `src/editor` |
| 前端 API、通用 UI、无业务工具 | `src/shared` |
| 后端业务接口 | `server/src/modules/<domain>` |
| schema 类型、迁移、校验 | `packages/lowcode-schema` |
| 数据模型变化 | `server/prisma` |

每个功能模块内部建议保持：

```text
<feature-or-domain>/
  model/             类型、展示字典、状态模型、纯业务规则
  ui/                可拆分复用的功能 UI 组件
  lib/               仅本模块使用的纯函数
  index.ts           可选公开出口
```

当前项目不要求马上把已有文件全部改成这个目录，但新增大功能应按这个形态落地。

示例：

```text
src/features/projects/
  ProjectDashboard.tsx       功能页面装配和接口交互
  model/display.ts           角色文案、颜色、审计动作文案
  model/types.ts             页面 props 和表单类型
```

编辑器内部也按同样思路渐进拆分：

```text
src/editor/stores/components.tsx          Zustand store 和 action 编排
src/editor/stores/component-tree.ts       组件树纯工具
src/editor/components/Header/index.tsx    顶部栏交互和 API 编排
src/editor/components/Header/schema.ts    schema 序列化
src/editor/components/Material/model.ts   物料面板内部模型和模板工具
src/editor/components/Material/builtinTemplates.ts 内置模板定义
src/editor/registry/factory.ts            物料注册公共工厂
src/editor/registry/types.ts              物料注册公共类型
src/editor/registry/configs/*             按分类拆分的物料配置
```

## 渐进拆分路线

1. 建立边界检查：`npm run check:architecture`。
2. 消除重复入口：新代码只从 `src/shared/api` 引入 API，`src/api` 保留兼容 re-export。
3. 拆大功能文件：优先拆 `ProjectDashboard`、`AdminDashboard`、`component-config` 和 `p3.tsx`。
4. 为每个大模块补 `index.ts` 作为公开出口，外部只从公开出口引用。
5. 等边界稳定后，再评估是否把仓库物理迁移为 `apps/web`、`apps/server`、`packages/schema`。

## 架构检查

运行：

```bash
npm run check:architecture
```

它会检查：

- 前端新代码是否绕回 `src/api` 旧兼容层。
- 前端是否直接引用后端或数据库文件。
- 后端是否直接引用前端文件。
- 共享 schema 包是否依赖前端或后端实现。
- `src/api` 是否保持 re-export-only。

完整提交前建议运行：

```bash
npm run check
```
