# UI 需求表达指南：如何让 Codex 写出你想要的界面

这份文档用于把“我想要这个 UI”转成 Codex 更容易执行和验收的工程任务。它适合编辑器面板、设置区、表单、工具栏、列表、弹窗、物料卡片等界面优化。

## 核心原则

不要只说“按某某产品改一下”或“更好看一点”。更好的方式是给出：

1. 当前截图
2. 目标截图
3. 明确差异
4. 可量化布局规格
5. 是否允许改组件结构
6. 验收方式

UI 任务最容易失败的地方，是只改 CSS 覆盖第三方组件内部结构。遇到 tab、表格、表单、复杂列表、事件编排卡片这类界面时，如果目标样式很明确，应该允许 Codex 改 JSX 结构，而不只是覆盖 Ant Design 的 class。

## 推荐提问模板

```md
我要调整这个 UI。

当前图：
<粘贴当前截图>

目标图：
<粘贴目标截图>

目标规格：
- 高度：
- 宽度/列数：
- 对齐方式：
- 间距：
- 边框：
- 背景：
- 文字颜色/字号/字重：
- 选中/hover/disabled 状态：
- 不要出现的样式：

实现要求：
- 可以改 JSX 结构 / 只能改 CSS
- 不要影响现有业务逻辑
- 不要改无关文案
- 改完用浏览器截图检查，不符合继续调

验收标准：
- 1.
- 2.
- 3.
```

## 描述布局时要说清楚的点

### 尺寸

说清高度、宽度、最小宽度、是否等分。

好例子：

```text
右侧 tab 高度 52px，三个 tab 等宽，每个 tab 占 1fr。
```

不够明确：

```text
tab 高一点，像 AMIS。
```

### 对齐

要明确水平和垂直两个方向。

好例子：

```text
每个 tab 的文字水平居中、垂直居中。
```

不够明确：

```text
居中一点。
```

### 状态

说清默认态、选中态、hover 态、禁用态。

好例子：

```text
默认态黑色文字；选中态蓝色文字，底部 2px 蓝线；hover 只有文字变蓝，不要背景块。
```

### 分隔和边界

说清边框出现在哪里。

好例子：

```text
tab 之间有 1px 竖线，竖线高度 14px，居中显示；tabbar 底部有 1px 浅灰边线。
```

## 明确“不要什么”

负面约束很重要。它能避免 Codex 做出看似合理但不是你想要的设计。

常见写法：

```text
不要胶囊背景。
不要圆角选中块。
不要左对齐。
不要让 tab 宽度跟随文字。
不要使用渐变、阴影、发光。
不要把面板做成卡片套卡片。
```

## 什么时候要求改 JSX 结构

如果出现这些情况，直接允许改结构：

- 第三方组件内部 DOM 影响布局，例如 Ant Design Tabs 无法按预期撑满。
- 目标 UI 是强布局规则，例如三等分、拖拽列表、事件编排卡片。
- 需要严格控制 active 底线、分隔线、工具按钮位置。
- CSS 覆盖越写越多，但结果仍然不稳定。

推荐说法：

```text
不要强行覆盖 Ant Design 内部 class。如果默认组件结构不适合目标效果，可以改成自定义 JSX 结构，但保留原有交互逻辑。
```

## 让 Codex 先写规格再动手

当你对效果要求很明确，可以要求 Codex 先输出实现规格。

```text
先不要改代码。先用盒模型描述你准备怎么实现：display、height、grid/flex、border、active 状态、hover 状态。等我确认后再改。
```

这样可以提前发现理解偏差。

## 验收要具体

样式任务不能只靠 `lint` 和 `build`。最好要求截图验收。

推荐验收要求：

```text
改完后启动本地页面，用浏览器截图检查右侧面板。
如果截图里文字没有居中、tab 没有等宽、底线位置不对，就继续修。
最后告诉我跑了哪些命令，以及哪里没有完成截图验证。
```

## 实战示例：右侧设置面板 Tab

较差表达：

```text
这里的样式按 AMIS 改一下。
```

较好表达：

```text
当前图：
<当前截图>

目标图：
<AMIS 截图>

右侧 tab 要完全参考 AMIS：
- 高度 52px
- 三个 tab 等宽
- 每个 tab 内容水平垂直居中
- tab 之间有 1px 竖线
- 选中项蓝色文字
- 选中项底部有 2px 蓝线
- 不要胶囊背景
- 不要左对齐
- 中间 tab 文案显示为“外观”

不要继续覆盖 Ant Design Tabs。如果 AntD Tabs 不好控制，就改成自定义 grid tabbar，但保留属性/外观/事件切换逻辑。

改完后跑 lint 和 build，并尽量用浏览器截图验收。
```

## 实战示例：属性表单

```text
当前属性面板看起来很松散，label 和输入框脱节。

目标：
- 使用紧凑属性检查器布局
- label 在输入框上方，距离输入框 4-6px
- 每个表单项间距 12px
- 输入框高度 32px
- 折叠组标题高度 36px
- 内容区 padding 为 12px
- 不要横向 label 宽度占位
- 不要出现大块空白

可以调整 Form 的 CSS 和局部 JSX class，不要改 setter 数据结构和组件属性写入逻辑。
```

## 实战示例：事件动作卡片

```text
我要把事件面板改成类似 AMIS 的动作编排列表。

目标：
- 每个事件是一条 40px 高的标题栏
- 标题左对齐，操作按钮右对齐
- 展开后动作卡片在浅灰背景中
- 动作卡片左侧有拖拽手柄，主标题和说明上下排列
- 右侧工具按钮固定宽度，不随文字移动
- 空状态显示一个主按钮

不要改事件 schema，不要改运行时动作执行逻辑，只改视觉结构和 class。
```

## 给 Codex 的一句话强化指令

如果你觉得某次 UI 要求很重要，可以在结尾加：

```text
这是视觉还原任务，不要只做到功能可用。请优先保证盒模型、对齐、间距和状态符合上面的规格；如果现有组件结构阻碍实现，请改结构而不是堆 CSS 覆盖。
```

## 最小可用清单

每次 UI 任务至少给这 6 项：

- 当前图
- 目标图
- 3-8 条硬规格
- 3-5 条不要项
- 是否允许改 JSX
- 截图验收要求

给得越像验收清单，Codex 写出来越接近你想要的 UI。
