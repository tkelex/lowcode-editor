## Context

编辑器当前已经有 `editor-interaction-styling` 主规格，覆盖画布、右键菜单、设置面板、物料样式和回归验证。上一轮修复集中处理了右键菜单重新定位、样式即时生效、px 数值输入、恢复默认样式，以及 Button/Input/Select 等控件类物料的样式分发。

这次变更不是引入新架构，而是把“页面类 bug 能否由自动化巡检发现”落到执行流程：先用代码扫描找出仍然共享 `commonStyleSetters` 但编辑态可能只把 `styles` 作用到外壳的物料，再用 Playwright 验证用户路径。发现真实问题后优先补 e2e，再做最小实现修复。

## Goals / Non-Goals

**Goals:**

- 覆盖画布滚动、选中、右键菜单连续重定位等核心交互。
- 覆盖设置面板样式输入的即时反馈、焦点保持、px 后缀和恢复默认。
- 覆盖控件类物料的真实控件样式生效，尤其是 Button、Input、Select 以外的相似组件。
- 将发现的问题写成可重复运行的 e2e 回归。

**Non-Goals:**

- 不重写编辑器布局、不重做设置面板信息架构。
- 不改变物料 schema、事件 action schema 或接口契约。
- 不把所有视觉差异做成截图测试；本次优先使用可维护的 DOM/CSS 断言。

## Decisions

- 使用现有 `npm run test:e2e:editor` 作为巡检承载点。原因是该命令已经专门针对编辑器回归，便于开发者在 UI 修复后快速复测；相比新增测试命令，维护成本更低。
- 用精确 CSS 断言验证“真实控件生效”。例如对控件内部 `button`、`input`、`.ant-select-selector`、`.ant-switch` 等节点断言 padding、font-size、color 或 background，而不是只检查外层 `[data-component-id]`。
- 对控件类物料沿用 `splitControlStyles` 的分发策略。尺寸和定位类样式保留在编辑器外壳，颜色、字体、内边距等视觉样式进入真实控件，避免选中框尺寸和控件视觉混在一起。
- 巡检发现问题时先补测试再修实现。这样能保证缺陷可复现，并让修复是否真正覆盖用户路径变得明确。

## Risks / Trade-offs

- 测试依赖 Ant Design DOM 结构可能随版本变化而需要维护。缓解方式：只断言稳定的公开类名或真实表单元素，并把断言范围限制在编辑器回归文件。
- 过多视觉断言可能导致 e2e 变慢。缓解方式：复用现有 mock 页面，在单个用例中覆盖同类物料，不引入截图比对。
- 物料差异较大，不能机械地把所有样式都塞到真实控件。缓解方式：容器类组件继续使用外壳样式；控件类组件只拆分尺寸定位与视觉样式。
