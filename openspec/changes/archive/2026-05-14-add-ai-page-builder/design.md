## Context

当前项目已经形成低代码页面编辑闭环：页面 schema 通过 `packages/lowcode-schema` 迁移和校验，进入 `useComponetsStore` 组件树，在编辑器内拖拽、设置、预览、保存、版本化和发布。物料能力集中在 registry，运行态事件动作和数据源由 Preview 链路执行。公开发布页必须继续禁止自定义 JS 执行。

用户希望项目最终方向接近 amis。amis 的价值不只是物料数量，而是 schema 驱动、CRUD 脚手架、数据源构建器、事件动作和编辑器插件体系。当前项目不应直接替换为 amis，而应先把 AI 作为 schema 生成层接入现有低代码协议。

## Goals / Non-Goals

**Goals:**

- 让用户通过自然语言生成整页或局部页面区块，并继续使用现有编辑器精修。
- 让 AI 输出普通低代码组件树，而不是 React 源码或新的运行时格式。
- 在服务端调用模型，保护模型密钥，并复用项目权限、审计和错误处理边界。
- 对 AI 输出执行白名单、组件树、父子关系、事件动作和安全校验。
- 支持基于页面描述、接口说明、响应示例或数据源模型生成后台类页面草稿。
- 与正在推进的 `add-data-model-crud-generation` 保持协同：AI 可作为输入补全器或编排器，CRUD schema 仍由确定性生成器负责。

**Non-Goals:**

- 不生成任意 React/Vue/HTML 源码供用户直接执行。
- 不让前端直接持有模型 API key。
- 不在第一版实现通用 Agent 自动操作编辑器、多轮自主发布或自动访问外部系统。
- 不默认允许 AI 生成公开发布页可执行的 custom JS。
- 不替换现有组件树、Preview、保存、版本和发布架构。
- 不把 amis 作为运行时直接嵌入当前项目；只借鉴其 schema、CRUD、插件和数据源设计。

## Decisions

### 1. AI 输出低代码 schema，而不是应用代码

AI 接口的核心输出为 `{ components, summary, warnings, assumptions }`，其中 `components` 必须符合当前 `LowcodeComponentSchema[]`。前端确认后调用 `setComponents` 或局部插入逻辑写入现有组件树。

原因：生成结果可以继续拖拽编辑、保存、回滚和发布，也能复用现有 `migratePageSchema` 与 `validateComponentTree`。

备选：AI 生成 React 代码或 amis schema。已放弃，因为会引入第二套运行时和保存协议，难以复用当前编辑器能力。

### 2. 后端统一调用模型并做权限、限流和审计

新增服务端 AI 模块，例如 `server/src/modules/ai`。接口需要校验当前用户对 project/page 的访问权限；记录生成请求摘要、模型、耗时、结果状态和失败原因；模型 key 只通过服务端环境变量配置。

原因：AI 生成会消耗额度并涉及用户业务描述，必须由后端统一做鉴权、审计、限流和错误处理。

备选：前端直接调用模型 API。已放弃，因为会暴露密钥，也绕过项目权限和审计。

### 3. 使用两阶段输出：结构化意图 -> schema

第一阶段让模型输出页面意图：页面类型、模块、字段、数据源、交互、布局说明。第二阶段将意图映射到当前物料 schema；第一版可由同一个接口内部完成，但代码结构应保留边界。

原因：直接让模型生成最终组件树容易出现非法父子关系、遗漏 `props` 或使用不存在物料。意图层更适合解释、预览和修复。

备选：只提示模型输出完整 JSON。可作为 MVP 内部实现，但必须搭配严格校验和修复循环。

### 4. 生成器优先使用确定性规则，AI 负责补全和编排

对于 CRUD 页面，AI 应先产出数据源模型、字段映射和页面生成选项，再交给 `add-data-model-crud-generation` 中的 CRUD schema 生成器生成组件树。没有数据接口的静态页面和仪表盘区块，可以由 AI 直接生成受限物料树。

原因：CRUD 的字段、接口和事件动作更适合由确定性生成器产出，避免模型随意拼接 HTTP action。

备选：AI 直接生成所有 CRUD 表格、表单和事件动作。风险较高，第一版只作为修复兜底，不作为主路径。

### 5. AI 输出必须经过校验、修复和用户确认

服务端或共享层提供 `validateAiGeneratedComponents`，至少执行：

- JSON 格式校验。
- 物料名称必须存在于 schema registry。
- `Page` 根节点、`id`、`parentId` 和父子关系合法。
- 事件动作类型和字段在白名单内。
- 禁止默认生成 `custom` action；如后续允许，必须仅限编辑器预览且显式提示。
- URL、HTTP method、headers、body 等字段符合现有 HTTP action 规则。

校验失败时可触发一次模型修复或确定性修复；仍失败则返回错误和可读原因，不写入编辑器。

### 6. 前端采用确认式写入

AI 面板提供输入、生成中、结果预览、警告、错误和确认写入状态。写入模式包括：

- 替换整页：用生成 Page schema 替换当前组件树。
- 插入当前组件：将生成片段插入当前选中且可接收子组件的容器。
- 创建新页面：在项目后台或页面列表中基于生成 schema 创建新 Page。

原因：AI 生成具有不确定性，用户必须能先检查结果再决定是否覆盖当前工作。

## Risks / Trade-offs

- [AI 输出不稳定] -> 使用结构化 JSON schema、物料白名单、校验失败修复循环和用户确认。
- [生成结果看起来可用但事件不可运行] -> 第一版优先生成静态页面和简单 HTTP action；复杂动作必须暴露 warnings。
- [覆盖用户当前编辑内容] -> 默认不直接写入；替换整页前展示确认，并保留 undo/redo 或保存前快照。
- [模型密钥和额度风险] -> 后端持有 key，增加项目权限、限流、审计和可配置开关。
- [与 CRUD 生成器重复] -> 明确 AI 是输入补全和编排层，CRUD schema 由确定性生成器输出。
- [公开发布安全] -> 继续保持 `allowCustomJS=false`，AI 不得默认生成 custom action。

## Migration Plan

1. 新增共享 AI 类型和 schema 校验工具，不影响现有页面。
2. 新增后端 AI 模块和环境变量；默认可通过配置关闭。
3. 新增前端 AI 入口和确认式写入流程，初期只对 owner/editor 开放。
4. 接入 CRUD 生成器能力；当 `add-data-model-crud-generation` 未完成时，先支持静态页面和简单表单/表格草稿。
5. 补充单元测试、后端 build/API smoke、编辑器 e2e 和文档。

回滚策略：关闭 AI 入口或后端 AI 配置后，既有页面编辑、保存和发布不受影响；已生成并保存的页面仍是普通 Page schema，可继续编辑或删除。

- AI 面板入口放在编辑器顶部、左侧物料面板，还是作为右侧设置面板的独立页签？
- 第一版是否允许“修改当前页面”的 patch 模式，还是先只做“生成后确认写入”？
- 是否需要保存 AI 生成历史，便于用户回看 prompt、结果和恢复旧草稿？
