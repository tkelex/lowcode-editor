## Context

项目当前已经完成 AI 页面搭建基础能力：前端 `AiBuilderPanel` 收集 prompt、接口说明和响应示例，后端 `AiService` 通过 AI 网关或本地降级生成页面草稿，生成结果经过 `validateAiGeneratedComponents` 校验，用户确认后才替换整页或插入当前容器。这个能力解决了“从零生成草稿”，但缺少 agent 所需的持续上下文、工具使用、计划执行、校验反馈修复和多轮修改能力。

新的 agent 编排必须继续遵守项目边界：AI 只产出低代码 schema 或 schema patch；模型调用只能走后端；前端不保存或发送模型 API key；公开发布页禁止执行 custom JS；所有写入都必须经过现有组件树校验和用户确认。它还需要与后续数据源/CRUD 生成器协同，让确定性生成器负责可预测结构，让模型负责理解意图、补全上下文和组织修改。

## Goals / Non-Goals

**Goals:**

- 让 AI 能读取当前页面、选中组件、物料能力、数据源模型、历史对话和校验结果等受控上下文。
- 让后端编排 agent 计划和工具调用，并记录可审计的 run/session。
- 支持多轮修改当前页面，输出候选组件树或结构化 patch，并在用户确认后写入。
- 支持校验失败后的受控修复循环，避免非法物料、非法父子关系、非法事件动作或 custom JS 写入。
- 前端展示 agent 的步骤、工具结果、warnings、assumptions、diff/预览和确认入口，让用户能理解并撤销。
- 对工具调用、模型输出、执行步数、上下文大小、权限和超时设定明确边界。

**Non-Goals:**

- 不实现通用浏览器自动操作、自动发布或自动访问任意外部系统。
- 不让 agent 直接执行 shell、数据库写入、任意 HTTP 请求或用户自定义 JS。
- 不生成 React/Vue/HTML 源码作为页面运行格式。
- 不在第一版实现完全自主长期任务队列；agent run 仍由用户在编辑器中显式触发。
- 不绕过现有保存、版本、发布、权限和审计流程。

## Decisions

### 1. 后端拥有 agent 编排，前端只负责会话 UI 和确认写入

agent run 由后端 `server/src/modules/ai` 负责创建、推进和审计。前端提交用户消息、目标范围和可选上下文偏好，后端加载项目/页面权限、裁剪上下文、调用模型、执行工具、校验输出并返回候选结果。

原因：模型 key、权限、审计、限流和工具白名单都必须在服务端统一控制。前端只展示状态并在用户确认后调用现有 store 写入，能避免模型直接操纵设计态数据。

备选方案是前端本地编排工具调用。已放弃，因为会暴露更多上下文和工具边界，也难以做可靠审计与限流。

### 2. agent 工具必须白名单化并按能力分层

第一版工具分为四类：

- 只读上下文工具：读取当前页面摘要、选中组件路径、可用物料、事件能力、数据源模型摘要。
- 生成工具：从意图生成 schema、从描述生成局部区块、调用 CRUD 生成器或降级静态结构。
- 修改工具：基于当前组件树产出 schema patch，或产出替换/插入候选树。
- 校验工具：运行 `validateAiGeneratedComponents`、组件树校验、事件动作安全校验和 patch 适用性校验。

每个工具使用结构化输入输出，必须声明是否可写。第一版所有可写工具都只产出候选结果，不直接持久化页面。

### 3. 使用“候选 patch 优先，整树替换兜底”的修改协议

多轮修改默认让 agent 输出结构化 patch，例如新增节点、更新 props/styles、移动节点、删除节点。patch 应包含目标组件 id、操作类型、原因和可展示摘要。复杂场景可返回完整候选组件树作为兜底，但写入前仍要 diff 和确认。

原因：patch 更适合“把按钮改成危险色”“给当前卡片加搜索表单”等局部修改，也更容易展示 diff 和撤销。整树替换用于大规模重构或旧模型无法稳定 patch 的情况。

### 4. 校验修复循环有硬边界

agent 输出候选结果后必须先校验。校验失败时，系统可以把精简错误反馈给模型进行有限次数修复，也可以执行确定性修复，例如补 parentId、重排 id、剔除非法 action。修复次数、总步数和耗时必须受配置限制。仍失败时只返回错误和可读解释，不写入编辑器。

原因：AI 输出不稳定，修复循环能提升成功率；硬边界能避免无限循环、费用失控和用户等待过久。

### 5. 上下文包要可解释、可裁剪、不可泄露密钥

上下文包应包含当前页面结构摘要、组件路径、选中组件、可用物料和 setters/events/methods 摘要、页面变量/数据源摘要、用户最新消息和必要历史。它不应包含模型 API key、JWT、完整审计日志、无关项目数据或未授权页面内容。

原因：agent 需要足够上下文才能正确修改页面，但上下文越大越容易泄露、超限和产生无关修改。

### 6. 用户确认是唯一写入闸门

agent 完成后返回候选 patch/组件树、摘要、风险、warnings、assumptions 和 diff/预览数据。用户点击确认后，前端才调用现有 store action 应用到当前页面；保存、版本和发布继续走既有流程。后续如果增加后端直接创建新页面，也必须复用项目权限和页面创建 API。

## Risks / Trade-offs

- [agent 修改范围过大] -> 默认以选中组件或用户指定范围为目标，展示 diff，确认前不写入。
- [工具调用结果不可靠] -> 工具输出结构化结果并强制校验，失败时返回可定位错误。
- [模型循环费用和耗时失控] -> 配置最大步数、最大修复次数、超时、上下文 token 预算和取消能力。
- [用户误以为 agent 已保存页面] -> UI 明确区分“候选修改已应用到编辑器”和“页面已保存到后端”。
- [安全边界被 prompt 注入绕过] -> 工具白名单、服务端权限校验、禁止任意代码执行、校验 custom action 和 URL/HTTP 规则。
- [与 CRUD 生成器重复] -> agent 负责理解和编排，CRUD 生成器负责确定性 schema 产出。

## Migration Plan

1. 先增加 shared agent 类型、patch 类型、工具调用结果类型和 patch 校验工具，不影响现有 AI 生成入口。
2. 在后端新增 agent orchestration service 和只读/校验工具 registry，保留现有 `generate` 接口兼容。
3. 将现有 AI 生成逻辑封装为 agent 可调用的生成工具，先支持单轮 run，再支持多轮 session。
4. 前端新增对话式 agent 面板，与现有 `AiBuilderPanel` 并存或逐步替换；默认仍要求用户确认后应用。
5. 补齐测试和文档后，再把“生成草稿”入口迁移到 agent run 之上。

回滚策略：关闭 agent 入口或后端 agent 配置后，现有 AI 页面搭建、手动编辑、保存和发布继续可用。已应用的 agent 修改仍是普通 Page schema，可继续编辑或通过版本回滚恢复。

## Open Questions

- 第一版 agent session 是否需要持久化完整对话历史，还是只在一次编辑器会话内保留？
- schema patch 是否采用项目自定义格式，还是引入标准 JSON Patch 并增加低代码语义扩展？
- agent 的流式状态是否需要第一版实现，还是先用轮询 run 状态满足可观测性？
- 工具调用审计是否只写入摘要，还是需要保存每次工具输入输出的脱敏快照？
