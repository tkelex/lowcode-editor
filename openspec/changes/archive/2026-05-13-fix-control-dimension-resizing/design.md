## Context

编辑器已经支持在外观面板中编辑 `width`、`height` 等 px 尺寸样式，并通过 `splitControlStyles` 将部分样式分配到编辑器外壳或真实 Ant Design 控件。这个策略解决了选中框定位与真实控件视觉样式混杂的问题，但当前对 `width`、`height` 的处理过于偏向外壳，导致 Button、Input 等控件在画布中可能只改变可选中区域或布局占位，真实控件本体没有同步变化。

低代码编辑器里，控件宽高应当是用户可配置能力。例外只应来自组件自身无法合理表达该尺寸，或 Ant Design 组件特性限制；这些例外需要明确处理，而不是表现为“输入了宽高但没有效果”。

## Goals / Non-Goals

**Goals:**

- 让按钮、输入框、下拉框等控件类物料的宽度和高度在画布编辑态真实可见。
- 保持选中框、拖拽根节点和实际控件尺寸对齐，不再出现“蓝色框变了、控件没变”的错觉。
- 保持 px 数字输入、清空删除样式、恢复默认样式等现有外观面板交互。
- 用 e2e 覆盖按钮和输入框等代表性控件的宽高修改路径。

**Non-Goals:**

- 不重新设计外观面板。
- 不改变持久化 schema，仍使用组件 `styles`。
- 不引入可视化拖拽 resize 手柄；本次只处理外观面板和 CSS 源码输入后的真实生效。
- 不强制所有容器类组件把尺寸传入子控件；容器类组件仍以外壳/容器自身为样式目标。

## Decisions

- 控件类物料的尺寸样式应同时影响外壳和真实控件。外壳负责选中框、拖拽和布局占位，真实控件负责用户可见的宽高；两者应在宽高类样式上保持一致。
- `splitControlStyles` 将保留“外壳样式”和“控件样式”的概念，但需要支持控件类物料获取共享尺寸样式，或提供明确的尺寸同步策略。
- 对 Button、Input、Select、Textarea、DatePicker、Upload、Popover、Notification 等以单个真实控件为主要视觉主体的物料优先修复。Switch、Rate 等天然尺寸受组件内部结构影响较大的物料，应至少保证外观面板中支持的尺寸字段不会表现为完全无效，并在测试里选择最稳定的代表组件。
- e2e 使用 DOM/CSS 断言而不是截图对比，验证外壳和真实控件节点都收到预期宽高，降低视觉测试维护成本。

## Risks / Trade-offs

- [Risk] 将宽高同时给外壳和真实控件可能改变部分 inline 组件的布局表现。→ Mitigation：优先处理控件类物料，并通过 e2e 覆盖 Button/Input/Select 等常用组件。
- [Risk] Ant Design 内部 DOM 对 `height` 的支持因组件不同而不同。→ Mitigation：对代表组件断言稳定节点；实现时针对不同组件选择真实可见的根控件节点。
- [Risk] 已有页面依赖“只改变外壳”的旧行为。→ Mitigation：这是编辑态样式真实生效修复，符合既有规格意图；不迁移 schema，仅改变渲染解释方式。
