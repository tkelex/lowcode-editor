## Context

项目是 Vite + React + TypeScript 低代码编辑器，核心风险集中在编辑器状态、物料注册表、编辑态/预览态渲染、事件动作运行时、项目保存加载和后端权限接口。近期已有多轮 UI 样式和交互优化，因此本次变更需要从“功能是否真的生效”出发，审查按钮、表单项、菜单、抽屉、设置器和预览行为是否已经完成实际接线。

审查结果应服务于后续修复，而不是只输出宽泛建议。每个问题都要能定位到代码位置，并尽量给出复现路径、用户影响、风险等级和建议修复方向。

## Goals / Non-Goals

**Goals:**

- 系统审查主要用户路径中是否存在无实际作用的 UI 控件、状态未写入、事件未触发、接口未调用或异常路径缺失。
- 识别编辑态和预览态不一致的问题，包括物料 dev/prod props、事件注入、容器拖拽、设置面板回填和 schema 迁移。
- 输出结构化审查报告，按严重程度排序，并给出可执行的修复任务建议。
- 对高风险发现补充最小验证建议，例如单元测试、E2E、smoke 或人工验收路径。

**Non-Goals:**

- 不在本变更中重构编辑器架构或重命名历史兼容 API。
- 不把所有发现一次性修复完；修复应按问题严重程度和影响范围拆分。
- 不新增大型自动化审计框架；优先用现有脚本、`rg`、测试和人工走查完成。
- 不改变公开发布页的安全边界，尤其不能让发布页执行 `custom` JS。

## Decisions

1. 以“工作流审查”作为主线，而不是按目录机械遍历。

   理由：无实际作用的控件通常出现在跨模块链路中，例如 UI 按钮存在但没有调用 store/API，或设置面板写入了 props 但 prod 物料没有消费。按用户工作流审查更容易发现断点。

   备选方案是只做静态未引用扫描。该方式速度快，但容易漏掉运行态行为不一致、旧 schema 兼容和权限接口问题。

2. 报告采用“发现优先”的交付格式。

   理由：用户明确想找 bug，因此报告应先列问题，再写背景说明。每条发现必须包含文件位置、触发条件、影响和建议修复方向，便于后续 `/opsx:apply` 或单独修复。

   备选方案是写完整测试计划。该方式有价值，但对当前目标来说不够直接。

3. 将审查范围分成静态扫描、关键链路走查和验证建议三层。

   理由：静态扫描用于快速发现未使用、空实现、TODO、按钮无 handler 等明显问题；关键链路走查用于确认编辑器实际行为；验证建议用于把发现转成可回归的质量门禁。

   备选方案是直接运行全部验证命令后看失败项。该方式只能发现已有测试覆盖的问题，不能覆盖“样式已做但行为没做”的情况。

4. 审查报告落在 `reports/functional-bug-audit.md`。

   理由：`reports/` 已是仓库中的报告类目录，适合保存一次性审查结果，不会混入产品需求文档或 OpenSpec 规范正文。

   备选方案是写到 OpenSpec change 目录。该方式便于归档，但报告作为项目现场审查结果，放在 `reports/` 更便于用户后续查看和追踪。

## Risks / Trade-offs

- 审查范围过大导致发现不聚焦 → 先覆盖编辑器核心路径和项目入口，再按严重程度扩展到后台与后端接口。
- 静态扫描误报较多 → 每条问题在报告前必须经过代码语义确认，不能只因为未引用或命名相似就判定为 bug。
- 无法在无完整本地服务状态下复现所有问题 → 对无法运行的场景明确记录验证限制，并给出后续复现命令。
- 审查只输出报告但不修复 → 在 `tasks.md` 中把高严重问题整理为后续修复队列，便于下一步执行。
